# /plan — 구현 계획 수립

## 절차

1. **요구사항 재확인**: 마스터(제이)의 요청을 자신의 말로 재정리하여 확인
2. **영향 범위 분석**: 변경이 영향을 미치는 팀/파일/DB 테이블 목록 작성
3. **위험 요소 파악**: 실투자 영향, OPS 안정성, 데이터 정합성 위험 평가
4. **단계별 계획**: 구현 순서를 단계별로 분해 (각 단계는 독립 커밋 가능 단위)
5. **테스트 전략**: 각 단계의 검증 방법 명시 (문법/소프트/하드 테스트)
6. **확인 대기**: 계획을 마스터에게 제시하고 승인 전까지 구현하지 않음

## 팀 제이 규칙

- 모든 구현은 DEV(맥북에어)에서 진행. OPS 직접 수정 금지.
- 실투자 영향이 있는 변경은 반드시 Shadow Mode 검증 포함
- 6대 원칙 준수: 자율과 통제, 감지와 판단 분리, 정합성 우선
- 비용 의식: LLM 호출 비용/빈도 영향 평가 포함

## 출력 형식

```
## 계획: [제목]
### 1. 요구사항
### 2. 영향 범위
  - 팀: [영향 팀 목록]
  - 파일: [변경 파일 목록]
  - DB: [영향 테이블/스키마]
### 3. 위험 요소
  - [위험1]: [심각도] / [대응 방안]
### 4. 단계별 구현
  - Step 1: [설명] → 검증: [방법]
  - Step 2: ...
### 5. 테스트 전략
```
