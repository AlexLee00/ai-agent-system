    [Node.js 63강] Lambda 서버리스 함수

    커피랑도서관 서현점 ・ 2026. 4. 13. 7:00
    URL 복사 통계
    본문 기타 기능

    ​

    Lambda 서버리스 함수는 서버를 직접 운영하지 않고도 이벤트 기반 백엔드 로직을 실행할 수 있게 해주는 실무 핵심 기술입니다.

    이번 63강에서는 Node.js 환경에서 Lambda를 왜 쓰는지, 어떤 구조에 적합한지, 서버 운영 방식과 무엇이 다른지 살펴봅니다.

    특히 배포 편의성만이 아니라 비용 구조, 확장성, 실행 시간 제약, 운영 관점까지 함께 이해하는 것이 중요합니다.

    ​

    지난 62강에서는 S3 파일 스토리지 연동을 통해 서버와 파일 저장 책임을 분리하는 구조를 함께 살펴보았습니다. 그렇다면 저장소를 분리한 다음, 실행 로직 자체도 서버에서 떼어낼 수는 없을까요. 오늘처럼 화창한 날이라 가볍게 걷기 좋은 분위기에는 생각도 조금 정리되는데, 그럴수록 서버를 늘리고 줄이는 방식보다 필요한 순간에만 실행되는 구조가 더 또렷하게 보이곤 합니다. 최근 기술 뉴스들을 보다 보면 작은 모델과 정교한 구조가 실제 문제를 더 잘 해결하는 장면이 자주 보입니다. 이번 63강에서는 바로 그 흐름과 맞닿아 있는 Lambda 서버리스 함수를 차분하게 정리해보겠습니다.

    ​

    이번 강의의 메인 주제는 분명하게 Lambda 서버리스 함수입니다. 많은 분들이 서버리스라는 말을 처음 들으면 “서버가 아예 없다”는 뜻으로 오해하십니다. 하지만 실제로는 서버가 없는 것이 아니라, 서버를 직접 관리하지 않아도 되는 실행 모델에 가깝습니다. 애플리케이션 개발자는 함수 단위의 로직만 배포하고, 실행 환경 준비, 스케일링, 인프라 유지, 일부 모니터링 기반은 클라우드 제공자가 맡는 구조입니다. 그래서 Lambda를 이해하려면 단순히 함수 하나를 올리는 방법보다, 실행 단위가 서버에서 함수로 바뀌는 운영 모델의 차이를 먼저 이해하는 것이 중요합니다.

    ​

    최신 Node.js 릴리스 흐름을 보면 v25.9.0, v25.8.2 같은 현재 계열과 v24.14.1 LTS가 함께 운영되고 있습니다. 여기서 실무적으로 중요한 포인트는, Lambda 같은 서버리스 환경에서는 “최신 기능”보다 “런타임 지원 범위”와 “안정적 배포 가능성”이 더 중요하다는 점입니다. 실제 AWS Lambda는 지원하는 Node.js 런타임 버전이 정해져 있고, 로컬 개발 환경에서 쓰는 버전과 클라우드 실행 환경 버전이 다를 수 있습니다. 즉, 로컬에서 Node.js 최신 버전으로 잘 되던 코드가 Lambda 배포 환경에서는 예상과 다르게 동작할 수도 있습니다. 그래서 서버리스 환경에서는 Node.js 자체의 최신성보다, 대상 플랫폼이 공식 지원하는 런타임과 호환성을 먼저 확인하는 습관이 중요합니다.

    ​

    Node.js가 Lambda와 잘 맞는 이유는 분명합니다. 비동기 I/O 처리에 강하고, 짧은 요청 응답 로직이나 이벤트 기반 처리에 적합하며, 외부 서비스와의 연동 코드도 비교적 간결하게 구성할 수 있기 때문입니다. 예를 들어 이미지 업로드 후 후처리, 결제 웹훅 수신, 예약 이벤트 처리, 일정 시간에 맞춘 배치성 작업, 간단한 API 응답 생성 같은 기능은 Lambda 서버리스 함수로 잘 분리되는 대표적인 사례입니다. 반대로 장시간 연결 유지가 필요한 소켓 서버나 복잡한 프로세스 상태를 오래 들고 있어야 하는 구조는 Lambda에 덜 어울릴 수 있습니다.

    ​

    아래는 가장 단순한 Lambda 핸들러(handler, 클라우드가 호출하는 진입 함수) 예시입니다.

    exports.handler = async (event) => { return { statusCode: 200, body: JSON.stringify({ message: 'Lambda 실행 성공', input: event, }), }; };

    이 짧은 코드 안에도 Lambda의 핵심이 들어 있습니다. 서버를 띄우거나 포트를 열지 않습니다. 대신 이벤트를 받아 결과를 반환합니다. 즉, Express처럼 애플리케이션이 계속 떠서 요청을 기다리는 구조가 아니라, 필요할 때 함수가 실행되고 끝나는 구조입니다.

    ​

    조금 더 현실적인 예시를 보겠습니다. 환경 변수를 읽어 외부 설정과 함께 동작하는 함수는 다음처럼 작성할 수 있습니다.

    exports.handler = async () => { const stage = process.env.STAGE || 'dev'; return { statusCode: 200, body: JSON.stringify({ stage, now: new Date().toISOString(), }), }; };

    여기서 중요한 것은 상태(state)를 오래 들고 있지 않는 사고방식입니다. Lambda는 호출마다 독립적으로 실행될 수 있고, 언제든 새 실행 환경에서 시작될 수 있기 때문입니다. 그래서 “이전 요청에서 메모리에 올려둔 값이 다음에도 있겠지” 같은 기대는 위험합니다.

    ​

    Lambda에서 자주 같이 나오는 개념이 이벤트 소스입니다. API Gateway를 통해 HTTP 요청이 들어올 수도 있고, S3 업로드를 계기로 실행될 수도 있으며, EventBridge 스케줄에 의해 정해진 시간마다 실행될 수도 있습니다. 예를 들어 S3 객체 업로드를 감지하는 흐름은 다음처럼 이벤트를 받을 수 있습니다.

    exports.handler = async (event) => { const record = event.Records?.[0]; const bucket = record?.s3?.bucket?.name; const key = record?.s3?.object?.key; console.log('업로드 감지', { bucket, key }); return { ok: true }; };

    이처럼 Lambda 서버리스 함수는 “요청을 오래 받는 서버”라기보다 “특정 사건에 반응하는 코드 조각”으로 이해하는 편이 좋습니다. 이 차이를 이해하지 못하면 Express 서버를 그대로 Lambda에 옮기려다가 구조가 복잡해지기 쉽습니다.

    ​

    혹시 서버리스가 단순히 서버 비용을 아끼는 기술 정도로만 보이셨다면, 실제로는 요청 처리 방식 자체를 서버 중심에서 이벤트 중심으로 바꾸는 사고 전환이라는 점을 꼭 기억해두시면 좋겠습니다.

    ​

    보안 관점에서도 Lambda는 특징이 분명합니다. 서버 접근 권한을 직접 관리하는 부담은 줄어들 수 있지만, 대신 IAM 권한, 환경 변수 비밀값, 함수별 최소 권한 정책이 매우 중요해집니다. 예를 들어 하나의 Lambda 함수가 S3 읽기만 필요하다면, 그 함수에는 정말 그 권한만 주는 편이 좋습니다. 서버리스는 관리 포인트가 사라지는 것이 아니라, 관리 포인트가 인스턴스 관리에서 권한과 이벤트 설계 관리로 이동하는 것에 가깝습니다.

    ​

    또 하나 중요한 것은 콜드 스타트(cold start, 새 실행 환경이 준비되며 처음 실행될 때 생기는 지연)입니다. Node.js는 서버리스에서 비교적 빠른 편이지만, 함수 크기가 커지거나 초기화 로직이 무거우면 첫 응답이 느려질 수 있습니다. 예를 들어 거대한 라이브러리를 여러 개 불러오거나, 함수 시작 시점에 불필요한 네트워크 호출을 해버리면 체감 성능이 크게 떨어질 수 있습니다. 그래서 Lambda 서버리스 함수에서는 기능 구현뿐 아니라 초기화 비용을 얼마나 줄일 수 있는가도 중요한 성능 포인트가 됩니다.

    const crypto = require('node:crypto'); exports.handler = async () => { const requestId = crypto.randomUUID(); return { statusCode: 200, body: JSON.stringify({ requestId }), }; };

    이처럼 가능하면 가벼운 표준 라이브러리를 우선 활용하고, 꼭 필요한 의존성만 포함하는 습관이 서버리스 환경에서는 더 중요합니다.

    ​

    정리하면, 최신 Node.js 환경에서 Lambda 서버리스 함수는 단순 배포 편의 기술이 아닙니다. 실행 모델, 권한 설계, 확장 방식, 비용 구조, 성능 최적화 포인트까지 함께 바꾸는 아키텍처 선택입니다. 이번 63강에서는 바로 이 시각, 즉 “서버를 덜 만지는 법”이 아니라 “실행 구조를 함수 단위로 다시 설계하는 법”으로 Lambda를 이해해보시면 좋겠습니다.

    비즈니스 관점에서 Lambda 서버리스 함수는 “서버를 안 써도 된다”는 기술이 아니라, 트래픽 변동성과 운영 복잡도를 비용 구조에 맞게 다시 설계하는 방법에 가깝습니다. 전통적인 서버는 트래픽이 적어도 계속 비용이 나가고, 피크 타임을 대비해 여유 자원을 잡아둬야 하는 경우가 많습니다. 반면 Lambda는 호출이 있을 때만 비용이 발생하고, 이벤트성 작업이나 간헐적 API, 후처리 작업처럼 사용량 편차가 큰 기능에 특히 잘 맞습니다. 예를 들어 회원 가입 후 이메일 발송, 이미지 리사이징, 예약 알림 전송, 결제 웹훅 처리 같은 로직은 24시간 서버를 띄워두는 것보다 Lambda 함수로 분리했을 때 비용과 운영 부담을 함께 줄이기 쉽습니다. 저는 실무에서 Lambda 도입을 단순한 클라우드 유행이 아니라, “항상 켜둘 필요가 없는 기능을 얼마나 잘 분리하느냐”의 문제로 보는 편입니다. 결국 서버리스의 핵심은 서버를 없애는 것이 아니라, 상시 운영이 필요하지 않은 책임을 서버 밖으로 꺼내는 데 있습니다.

    ​

    이제부터는 본격적으로 Lambda 서버리스 함수의 핵심 개념을 정리해보겠습니다. 앞에서 Lambda를 비용 구조와 운영 방식의 관점에서 설명드렸다면, 이번에는 실제로 어떤 개념을 이해해야 Lambda를 실무에서 제대로 쓸 수 있는지 차근차근 짚어보겠습니다. Lambda 서버리스 함수는 단순히 “서버를 안 써도 되는 기술”이 아닙니다. 실행 단위를 서버에서 함수로 바꾸고, 운영 책임을 인스턴스 관리에서 이벤트 설계와 권한 설계로 옮기는 구조적 전환에 가깝습니다.

    ​

    많은 분들이 처음 서버리스를 접할 때 가장 헷갈려하는 부분은 이것입니다. “서버가 없다면 도대체 어디서 코드가 돌아가는가?” 실제로는 서버가 사라진 것이 아니라, 서버를 직접 띄우고 패치하고 감시하는 책임이 개발자 손에서 상당 부분 벗어난 것입니다. 즉, 개발자는 함수(handler, 클라우드가 호출하는 진입 함수)와 필요한 설정만 올리고, 실행 환경 준비와 확장 처리는 클라우드 제공자가 맡습니다. 그렇기 때문에 Lambda를 이해할 때는 “서버를 없앴다”보다 “서버 관리의 추상화 수준이 올라갔다”라고 이해하는 편이 더 정확합니다.

    ​

    Lambda 서버리스 함수의 핵심은 세 가지 축으로 이해하시면 좋습니다.

    ​

    1. 이벤트 기반 실행 구조입니다

    전통적인 서버 애플리케이션은 프로세스를 계속 띄워두고 요청을 기다립니다. 반면 Lambda는 특정 이벤트가 발생했을 때만 실행됩니다. HTTP 요청이 들어올 수도 있고, 스케줄 이벤트가 발생할 수도 있고, S3 업로드나 메시지 큐 이벤트가 들어올 수도 있습니다. 즉, “항상 대기 중인 서버”가 아니라 “필요한 순간에만 깨워지는 함수”에 가깝습니다.

    ​

    2. 상태를 오래 들고 가지 않는 구조입니다

    Lambda는 같은 실행 환경이 재사용될 수도 있지만, 언제든 새 환경에서 시작될 수도 있습니다. 그래서 메모리에 남아 있는 값을 다음 호출에서도 쓸 수 있다고 기대하면 위험합니다. 이 때문에 상태(state, 실행 중 유지되는 값)는 외부 저장소에 두고, 함수 자체는 가능하면 짧고 명확하게 끝내는 편이 좋습니다.

    ​

    3. 실행 시간과 자원에 제약이 있습니다

    Lambda는 영원히 실행되지 않습니다. 정해진 시간 안에 끝나야 하고, 메모리와 CPU도 설정 범위 안에서 움직입니다. 이 제약은 불편해 보일 수 있지만, 반대로 함수의 책임을 작게 나누고, 과도하게 비대해지는 것을 막아주는 장치가 되기도 합니다.

    ​

    가장 기본적인 Lambda 함수는 아래처럼 생겼습니다.

    exports.handler = async (event) => { return { statusCode: 200, body: JSON.stringify({ message: 'Lambda 실행 성공', input: event, }), }; };

    이 코드는 단순하지만 Lambda의 핵심을 잘 보여줍니다. 포트를 열지 않고, 서버를 listen 하지도 않습니다. 대신 입력 이벤트(event, 호출 시 전달되는 데이터)를 받아 결과를 반환합니다. 즉, Express처럼 앱 전체가 살아 있는 구조가 아니라, 함수 하나가 호출되고 끝나는 구조입니다.

    ​

    조금 더 현실적인 예시를 보겠습니다. 환경 변수(environment variable, 배포 환경에서 주입되는 설정값)를 읽는 코드는 다음처럼 간단합니다.

    exports.handler = async () => { const stage = process.env.STAGE || 'dev'; return { statusCode: 200, body: JSON.stringify({ stage, now: new Date().toISOString(), }), }; };

    이때 중요한 것은 환경 변수에 비밀값이나 외부 서비스 키가 들어갈 수 있다는 점입니다. 그래서 Lambda를 설계할 때는 코드뿐 아니라 권한과 비밀 관리도 같이 보셔야 합니다. 특히 제가 운영하는 ai-agent-system에서 여러 팀의 자동화 흐름을 점검하다 보면, 반복되는 에러 로그보다 더 위험한 것이 설정과 비밀값 관리가 느슨해지는 순간이라는 점을 자주 체감합니다. 덱스터 점검에서 secrets.json 관련 보안 경고가 반복적으로 올라오거나, 장기 적체된 배치 처리 문제와 에러 로그가 함께 보였던 경우를 떠올리면 더 분명합니다. 서버리스라고 해서 운영 리스크가 사라지는 것이 아니라, 리스크의 모양이 바뀌는 것입니다.

    ​

    Lambda에서 매우 중요한 개념 하나는 콜드 스타트(cold start, 새 실행 환경이 처음 준비되며 생기는 초기 지연) 입니다. 함수가 자주 호출되면 기존 실행 환경이 재사용될 수도 있지만, 오랜만에 호출되거나 갑자기 트래픽이 늘어나면 새 실행 환경이 생성됩니다. 이때 초기화 비용이 크면 첫 응답이 느려질 수 있습니다. 예를 들어 함수 바깥에서 거대한 라이브러리를 여러 개 불러오거나, 시작하자마자 불필요한 네트워크 호출을 하면 콜드 스타트가 길어집니다. 그래서 Lambda에서는 “기능이 되느냐” 못지않게 “초기화가 얼마나 가벼운가”도 중요합니다.

    const crypto = require('node:crypto'); exports.handler = async () => { const requestId = crypto.randomUUID(); return { statusCode: 200, body: JSON.stringify({ requestId }), }; };

    이처럼 가능하면 가벼운 표준 라이브러리를 활용하고, 함수 시작 시 꼭 필요한 작업만 하도록 설계하는 편이 좋습니다.

    ​

    또 하나 중요한 개념은 이벤트 소스(event source, 함수를 깨우는 입력 주체) 입니다. Lambda는 혼자 존재하는 것이 아니라, 거의 항상 어떤 이벤트와 연결됩니다. 대표적인 예시는 다음과 같습니다.

    ​

    1. API Gateway를 통한 HTTP 요청

    2. S3 파일 업로드 이벤트

    3. EventBridge를 통한 정시 실행

    4. SQS 같은 큐 메시지 처리

    5. DynamoDB 스트림 같은 데이터 변경 이벤트

    ​

    예를 들어 S3 업로드를 감지하는 Lambda는 이런 구조로 이벤트를 받습니다.

    exports.handler = async (event) => { const record = event.Records?.[0]; const bucket = record?.s3?.bucket?.name; const key = record?.s3?.object?.key; console.log('S3 업로드 감지', { bucket, key }); return { ok: true }; };

    이 코드를 보면 Lambda는 “요청을 기다리는 서버”보다 “사건에 반응하는 처리기”에 가깝다는 점이 분명해집니다. 이 차이를 이해하지 못하면 기존 Express 서버 구조를 그대로 Lambda에 옮기려다가 함수가 너무 비대해지거나, 역할이 어정쩡해질 수 있습니다.

    ​

    혹시 서버리스가 단순히 서버 비용을 줄이는 도구처럼 보이셨다면, 실제로는 요청을 처리하는 사고방식 자체를 서버 중심에서 이벤트 중심으로 바꾸는 기술이라는 점을 꼭 기억해두시면 좋겠습니다.

    ​

    Lambda를 실무에서 잘 쓰려면 권한 설계도 함께 보셔야 합니다. 서버 한 대에 모든 권한을 몰아주는 구조와 달리, Lambda는 함수마다 필요한 권한만 따로 줄 수 있습니다. 예를 들어 어떤 함수는 S3 읽기만 필요하고, 다른 함수는 특정 데이터베이스 테이블 쓰기만 필요할 수 있습니다. 이처럼 최소 권한 원칙을 지키기 쉬운 것은 Lambda의 큰 장점입니다. 하지만 반대로 함수 수가 많아질수록 권한 구성이 복잡해질 수도 있습니다. 그래서 서버리스 아키텍처에서는 함수 코드만큼 IAM 정책(권한 정책) 설계도 중요합니다.

    ​

    실행 시간 제한도 중요한 포인트입니다. Lambda는 짧고 분명한 작업에 특히 잘 맞습니다. 이미지 리사이징, 이메일 발송, 웹훅 처리, 예약 알림, 간단한 배치 작업, 파일 후처리처럼 경계가 분명한 작업은 Lambda와 잘 맞습니다. 반대로 오랜 연결 유지, 복잡한 세션 상태 유지, 무거운 연산을 장시간 반복해야 하는 작업은 다른 구조가 더 적합할 수 있습니다. 즉, Lambda는 만능이 아니라 짧고 명확하게 분리할 수 있는 작업에 강한 도구입니다.

    ​

    제가 운영하는 ai-agent-system에서도 이런 감각이 중요했습니다. 반복 에러 로그나 장기 적체가 생기는 흐름을 보면, 대개 “계속 살아 있어야 하는 책임”과 “필요할 때만 실행되면 되는 책임”이 섞여 있는 경우가 많았습니다. 서버리스는 그 경계를 더 분명히 만들 수 있게 도와줍니다. 항상 서버에 붙여둘 필요가 없는 작업을 함수로 떼어내면, 장애 범위도 줄고 운영 시야도 더 선명해집니다.

    ​

    정리하면 Lambda 서버리스 함수의 핵심 개념은 다음과 같습니다.

    ​

    1. 서버를 없애는 것이 아니라 서버 관리 책임을 추상화하는 것입니다.

    2. 항상 대기하는 서버가 아니라 이벤트에 반응하는 함수 구조입니다.

    3. 상태를 오래 들고 가지 않고 짧고 분명한 실행 단위로 나누는 것이 중요합니다.

    4. 콜드 스타트, 실행 시간, 권한 설계가 실무 품질을 크게 좌우합니다.

    5. Lambda는 작은 기능 하나를 빠르게 배포하는 도구이면서, 아키텍처 경계를 정리하는 도구이기도 합니다.

    ​

    저는 Lambda를 제대로 이해하면 이후 API Gateway, EventBridge, SQS, S3 트리거, 서버리스 배치 구조까지 이어서 볼 때도 훨씬 덜 흔들린다고 생각합니다. 결국 좋은 서버리스 설계는 함수 문법을 외우는 데서 나오지 않고, “무엇을 상시 서버에 둘 것인가, 무엇을 이벤트 함수로 분리할 것인가”를 잘 나누는 데서 나오기 때문입니다.

    기획 단계에서 Lambda 서버리스 함수를 설계할 때 가장 자주 놓치는 부분은 “이 기능을 서버리스로 만들 수 있는가”만 묻고 끝내는 점입니다. 실제로는 그보다 먼저 “이 기능이 항상 켜져 있어야 하는가”, “이 기능은 사건이 발생했을 때만 반응하면 충분한가”, “실행 시간이 짧고 경계가 분명한가”를 물어야 합니다. 예를 들어 예약 완료 알림, 결제 웹훅 처리, 파일 업로드 후 후처리, 정시 배치 알림 같은 기능은 Lambda와 잘 맞습니다. 반면 지속적인 연결 유지나 복잡한 상태 공유가 필요한 기능은 서버리스로 밀어 넣을수록 오히려 설계가 꼬일 수 있습니다.

    ​

    저는 기획 단계에서 기능 후보를 볼 때 다음 기준으로 먼저 나눠보는 편입니다. 첫째, 이벤트가 분명한가. 둘째, 실행 시간이 짧은가. 셋째, 외부 상태 저장소 없이 오래 버텨야 하지 않는가. 넷째, 함수별 최소 권한으로 쪼갤 수 있는가. 이 네 가지를 통과하면 Lambda 후보로 검토하기가 훨씬 수월해집니다. 결국 서버리스의 핵심은 유행을 따라가는 것이 아니라, 상시 서버가 필요하지 않은 책임을 얼마나 정확히 분리하느냐에 있습니다. 저는 그래서 Lambda를 기술 선택지라기보다, 책임을 더 작고 명확하게 나누는 설계 도구로 보는 편입니다.

    ​

    이제 이론과 기획 관점을 실제 코드로 연결해보겠습니다. 이번 섹션에서는 Lambda 서버리스 함수를 Node.js 기준으로 직접 구현하면서, 어떤 구조가 실무에서 오래 버티는지 살펴보겠습니다. 단순히 함수 하나를 올리는 예제가 아니라, 이벤트 입력 처리, 환경 변수 관리, 공용 응답 유틸, 외부 API 호출, 스케줄성 배치 함수까지 함께 보면서 Lambda다운 코드가 무엇인지 정리해보겠습니다. 오늘처럼 화창한 날에는 괜히 코드를 가볍게 보고 싶어지기도 하지만, 서버리스는 처음 구조를 단정하게 잡아둘수록 나중에 운영이 훨씬 편해지는 영역입니다.

    ​

    먼저 전제부터 분명히 하겠습니다. 이번 실습은 실제 npm 패키지와 Node.js 표준 라이브러리만 사용합니다. 예시는 AWS Lambda의 Node.js 런타임을 가정하고, 외부 HTTP 호출은 Node.js 표준 fetch 를 사용하겠습니다. 즉, 불확실한 서드파티 HTTP 클라이언트를 따로 들여오지 않습니다.

    ​

    실무에서 Lambda 서버리스 함수를 작성할 때 가장 먼저 해야 할 일은 핸들러(handler, 클라우드가 호출하는 진입 함수)를 얇게 유지하는 것입니다. 핸들러 안에 모든 로직을 한꺼번에 넣기 시작하면 테스트도 어렵고, 재사용도 어렵고, 콜드 스타트 최적화도 점점 힘들어집니다.

    ​

    ## 1. 가장 기본적인 Lambda 핸들러와 공통 응답 유틸

    ​

    아래 코드는 API Gateway와 연결되는 가장 기본적인 Lambda 서버리스 함수 예시입니다. 입력을 받고, 검증하고, JSON 응답을 반환합니다.

    'use strict'; /** * JSON 응답을 생성합니다. * @param {number} statusCode HTTP 상태 코드 * @param {object} data 응답 본문 객체 * @returns {{statusCode: number, headers: object, body: string}} */ function jsonResponse(statusCode, data) { return { statusCode, headers: { 'Content-Type': 'application/json; charset=utf-8', }, body: JSON.stringify(data), }; } /** * Lambda 핸들러입니다. * @param {object} event API Gateway 이벤트 객체 * @returns {Promise<{statusCode: number, headers: object, body: string}>} */ exports.handler = async (event) => { const name = event?.queryStringParameters?.name || '익명'; return jsonResponse(200, { message: `안녕하세요, ${name}님`, }); };

    이 코드는 단순하지만 중요한 원칙이 들어 있습니다.

    ​

    1. 응답 포맷을 공통 함수로 분리했습니다.

    2. 핸들러 내부에서는 이벤트를 읽고 바로 응답만 조립합니다.

    3. 로직이 커질 경우 별도 함수로 분리하기 쉬운 구조입니다.

    ​

    ### 안티패턴

    • 핸들러 안에서 검증, 비즈니스 로직, 응답 포맷을 모두 한 번에 처리하는 방식

    • console.log 만 잔뜩 남기고 구조화된 응답 형식이 없는 방식

    • 예외 발생 시 스택만 던지고 사용자 응답 형식을 정하지 않는 방식

    ​

    ### 권장 패턴

    • 입력 해석, 핵심 로직, 응답 생성 분리

    • JSON 응답 포맷 공통화

    • 핸들러는 얇게, 비즈니스 로직은 별도 함수로 분리

    ​

    ## 2. 환경 변수 검증과 설정 로딩 분리

    ​

    Lambda 서버리스 함수는 짧고 독립적이어야 하지만, 설정값은 오히려 더 엄격하게 다뤄야 합니다. 환경 변수 누락은 서버리스에서 굉장히 흔한 장애 원인입니다. 특히 배포 환경이 dev, stage, prod 로 갈라질수록 더 그렇습니다.

    'use strict'; /** * 필수 환경 변수를 읽고 검증합니다. * @returns {{ stage: string, webhookUrl: string }} */ function getConfig() { const stage = process.env.STAGE || 'dev'; const webhookUrl = process.env.WEBHOOK_URL; if (!webhookUrl) { throw new Error('WEBHOOK_URL 환경 변수가 누락되었습니다.'); } return { stage, webhookUrl }; } /** * 실제 비즈니스 로직 예시 함수입니다. * @returns {Promise<object>} */ async function getRuntimeInfo() { const config = getConfig(); return { stage: config.stage, now: new Date().toISOString(), }; } /** * Lambda 핸들러입니다. * @returns {Promise<{statusCode: number, headers: object, body: string}>} */ exports.handler = async () => { try { const result = await getRuntimeInfo(); return { statusCode: 200, body: JSON.stringify(result), }; } catch (error) { console.error('Lambda 실행 실패', { message: error.message }); return { statusCode: 500, body: JSON.stringify({ message: '서버 설정 오류' }), }; } };

    ### 안티패턴

    • 환경 변수를 함수 곳곳에서 직접 읽는 방식

    • 필수 환경 변수 누락 여부를 전혀 확인하지 않는 방식

    • 설정값이 잘못되어도 500 에러로만 뭉뚱그리는 방식

    ​

    ### 권장 패턴

    • 설정 로딩 함수 분리

    • 필수값 검증 명시

    • 로그에는 내부 원인을 남기되, 외부 응답은 정제

    ​

    이 차이는 작아 보여도 운영 단계에서 원인 파악 속도를 크게 바꿉니다.

    ​

    독자 소통 브릿지

    혹시 서버리스는 코드만 짧게 쓰면 된다고 생각하셨다면, 실제로는 환경 변수와 권한 설정이 더 자주 문제를 만든다는 점을 꼭 기억해두시면 좋겠습니다. Lambda는 서버가 안 보이는 만큼 설정 실수가 더 늦게 눈에 띄기도 합니다.

    ​

    ## 3. 외부 API를 호출하는 Lambda 서버리스 함수

    ​

    Lambda 서버리스 함수는 웹훅 처리, 알림 전송, 외부 API 통신 같은 작업과 특히 잘 맞습니다. 아래 예시는 외부 API를 호출해 데이터를 받아 단순 가공 후 반환하는 함수입니다. Node.js 표준 fetch 를 사용하므로 추가 HTTP 패키지가 필요 없습니다.

    'use strict'; /** * 외부 API에서 JSON 데이터를 가져옵니다. * @param {string} url 호출할 URL * @returns {Promise<object>} */ async function fetchJson(url) { const response = await fetch(url, { method: 'GET', headers: { 'Accept': 'application/json', }, }); if (!response.ok) { throw new Error(`외부 API 호출 실패: ${response.status}`); } return response.json(); } /** * Lambda 핸들러입니다. * @returns {Promise<{statusCode: number, body: string}>} */ exports.handler = async () => { try { const data = await fetchJson('https://api.github.com'); return { statusCode: 200, body: JSON.stringify({ current_user_url: data.current_user_url, repository_url: data.repository_url, }), }; } catch (error) { console.error('외부 API 처리 실패', { message: error.message }); return { statusCode: 502, body: JSON.stringify({ message: '외부 연동 중 오류가 발생했습니다.', }), }; } };

    이 구조에서 중요한 점은 다음과 같습니다.

    ​

    1. 외부 호출 함수를 분리해 테스트와 재사용이 쉬워집니다.

    2. response.ok 검사를 통해 실패를 조기에 잡습니다.

    3. API 응답 전체를 그대로 내보내지 않고 필요한 데이터만 선택합니다.

    ​

    ### 안티패턴

    • 핸들러 안에서 fetch 를 여러 번 중첩 호출하는 방식

    • 실패 상태 코드를 확인하지 않고 바로 response.json() 하는 방식

    • 외부 API 전체 응답을 그대로 노출하는 방식

    ​

    ### 권장 패턴

    • 호출 함수 분리

    • 상태 코드 검증

    • 반환 데이터 최소화

    ​

    ## 4. EventBridge 스케줄 기반 Lambda 배치 함수

    ​

    Lambda 서버리스 함수가 특히 강한 영역 중 하나는 “항상 켜둘 필요는 없지만 정해진 순간 실행되어야 하는 작업”입니다. 예를 들어 정시 통계 집계, 예약 알림, 오래된 데이터 정리 같은 작업입니다. 아래 코드는 스케줄 이벤트를 받아 실행되는 단순 배치 함수 예시입니다.

    'use strict'; const fs = require('node:fs/promises'); const path = require('node:path'); /** * 임시 배치 결과를 파일에 기록합니다. * 실제 운영에서는 S3, DB, 로그 시스템 등으로 대체될 수 있습니다. * @param {object} payload 기록할 데이터 * @returns {Promise<void>} */ async function writeBatchResult(payload) { const outputPath = path.join('/tmp', 'batch-result.json'); await fs.writeFile(outputPath, JSON.stringify(payload, null, 2), 'utf8'); } /** * 배치 로직을 수행합니다. * @param {object} event 스케줄 이벤트 * @returns {Promise<object>} */ async function runBatch(event) { const result = { triggerTime: event?.time || new Date().toISOString(), processedAt: new Date().toISOString(), status: 'ok', }; await writeBatchResult(result); return result; } /** * Lambda 핸들러입니다. * @param {object} event EventBridge 이벤트 * @returns {Promise<{statusCode: number, body: string}>} */ exports.handler = async (event) => { try { const result = await runBatch(event); return { statusCode: 200, body: JSON.stringify(result), }; } catch (error) { console.error('배치 실행 실패', { message: error.message }); return { statusCode: 500, body: JSON.stringify({ message: '배치 실행 실패' }), }; } };

    여기서 /tmp 경로를 쓴 이유는 Lambda 실행 환경에서 임시 파일 저장이 허용되는 표준 위치이기 때문입니다. 다만 이 저장소는 영구 보관용이 아니므로, 중요한 데이터는 외부 저장소로 보내야 합니다.

    ​

    ### 안티패턴

    • 장시간 돌아가는 배치를 Lambda 하나에 몰아넣는 방식

    • /tmp 를 영구 저장소처럼 믿는 방식

    • 실패 재시도 전략 없이 한 번 호출로 끝내는 방식

    ​

    ### 권장 패턴

    • 작은 배치 단위로 나누기

    • /tmp 는 임시 용도로만 사용

    • 실패 가능성을 전제로 로그와 재호출 흐름 설계

    ​

    ## 5. Lambda 아키텍처를 나눌 때의 기준

    ​

    실무에서 Lambda 서버리스 함수가 복잡해지는 이유는 대부분 “작은 함수로 나누지 못해서”입니다. 아래 기준으로 함수 경계를 나누면 훨씬 안정적입니다.

    ​

    1. 이벤트 종류가 다르면 함수도 분리합니다.

    • HTTP 요청 처리

    • 스케줄 배치

    • 파일 업로드 후처리

    ​

    2. 권한 범위가 다르면 함수도 분리합니다.

    • S3 읽기 전용

    • 외부 웹훅 전송

    • DB 쓰기 함수

    ​

    3. 실패했을 때 영향 범위가 다르면 함수도 분리합니다.

    • 알림 실패가 예약 생성 실패까지 번지지 않게

    • 통계 집계 실패가 실시간 API를 막지 않게

    ​

    이 원칙은 서버리스 아키텍처의 핵심입니다. 서버리스는 함수가 작을수록 이해하기 쉽고, 배포 단위도 명확하고, 권한도 최소화하기 쉽습니다. 반대로 “작은 서버 하나를 Lambda로 옮긴 것 같은 구조”가 되면 장점이 급격히 사라집니다.

    ​

    제가 운영하는 여러 자동화 흐름을 보면서도 비슷한 감각을 자주 느낍니다. 반복 오류 로그가 쌓이거나 장기 적체가 생기는 흐름은 대부분 책임 경계가 흐린 경우가 많았습니다. Lambda 서버리스 함수는 그 경계를 더 선명하게 그을 수 있게 도와줍니다. 항상 켜둘 필요가 없는 작업을 떼어내고, 짧고 분명한 이벤트 단위로 다시 나누는 것이 핵심입니다.

    ​

    정리하면, 실무 코드에서 좋은 Lambda 서버리스 함수는 아래 기준을 만족해야 합니다.

    ​

    1. 핸들러가 얇다

    2. 설정 검증이 분리되어 있다

    3. 외부 호출과 핵심 로직이 분리되어 있다

    4. 실패 상황을 고려한 응답 구조가 있다

    5. 함수 경계가 이벤트와 권한 기준으로 명확하다

    ​

    저는 이 기준이 잡혀 있어야 이후 API Gateway, EventBridge, SQS, Step Functions 같은 확장 주제로 넘어갈 때도 훨씬 덜 흔들린다고 생각합니다. 결국 좋은 서버리스 코드는 짧은 코드가 아니라, 책임이 분명한 코드이기 때문입니다.

    Lambda 서버리스 함수 코드의 비즈니스 가치는 “배포가 쉽다”에서 끝나지 않습니다. 함수 단위로 책임을 분리하면 장애 범위를 줄일 수 있고, 호출 기반 비용 구조 덕분에 상시 서버 운영비를 줄일 수 있으며, 기능별 최소 권한 설계가 쉬워져 보안 위험도 낮출 수 있습니다. 예를 들어 웹훅 처리, 예약 알림, 파일 후처리 같은 기능을 Lambda로 떼어내면, 해당 작업은 필요할 때만 실행되고 실패 영향도 국소화됩니다. 이것은 단순한 코드 스타일 차이가 아니라 운영 전략 차이입니다.

    ​

    저는 실무에서 좋은 Lambda 코드를 “작은 함수 몇 개”로 보지 않습니다. 오히려 상시 운영이 필요하지 않은 책임을 정확히 꺼내고, 비용과 장애 범위를 함께 줄이는 설계 도구로 봅니다. 핸들러 하나를 가볍게 유지하는 습관, 환경 변수와 권한을 명확히 분리하는 습관, 외부 연동 실패를 전제로 로그와 응답 구조를 갖추는 습관은 모두 비즈니스 가치를 만듭니다. 결국 Lambda 서버리스 함수 코드는 인프라 비용, 배포 속도, 보안 수준, 장애 복구 속도를 동시에 좌우하는 운영 코드입니다. 서비스 규모가 커질수록 그 차이는 더 분명하게 드러납니다.

    ​

    Lambda 서버리스 함수를 실무에 붙이기 시작하면 가장 먼저 체감하는 것은 “코드는 짧아졌는데 에러 해석은 오히려 더 어려워질 수 있다”는 점입니다. 서버를 직접 들여다보는 방식에서 이벤트 로그와 권한 정책, 트리거 흐름을 따라가는 방식으로 사고가 바뀌기 때문입니다. 예를 들어 함수 자체 코드는 맞는데 IAM 권한이 부족해서 실패할 수도 있고, 이벤트 소스 형식이 예상과 달라서 undefined 가 연쇄적으로 퍼질 수도 있습니다. 또 함수 실행은 끝났지만 실제 비즈니스 처리는 다운스트림 큐나 데이터 적체 때문에 늦어질 수도 있습니다. 그래서 Lambda를 다룰 때는 단순한 디버깅 감각보다, 에러를 층위별로 분류해서 읽는 사고 습관이 더 중요해집니다.

    ​

    저는 이런 흐름을 ACC-ERN처럼 이해하는 편입니다. 적응형 코드·클라우드 에러 인지망이라고 부를 만한 감각인데, 에러가 발생했을 때 “무엇이 틀렸나”만 보는 것이 아니라 “이 에러가 어느 층에서 왔는가”를 먼저 분류하는 방식입니다. 예를 들어 AccessDeniedException 은 권한 레이어, 타임아웃은 실행 시간과 외부 연동 레이어, 이벤트 필드 누락은 트리거 레이어, 반복 적체는 아키텍처 경계 설계 레이어 문제일 가능성이 큽니다. 실제로 제가 운영하는 자동화 흐름에서도 반복적인 에러 로그, 장기 적체, 전 팀 헬스 복구 문제는 각각 개별 오류처럼 보이지만, 결국은 책임 경계와 이벤트 해석 방식이 흐려졌을 때 함께 나타나는 경우가 많았습니다. 이런 패턴을 빠르게 읽어내는 힘이 서버리스 운영에서는 생각보다 훨씬 중요합니다.

    ​

    이런 작업을 오래 하다 보면 환경의 중요성도 무시하기 어려워집니다. 저는 커피랑도서관 분당서현점 같은 공간이 이런 설계와 디버깅 작업에 특히 잘 맞는다고 느끼는 편입니다. 단순히 조용하기 때문만은 아닙니다. Lambda 같은 서버리스 구조는 로그 한 줄, 환경 변수 하나, 이벤트 JSON 구조 하나를 오래 들여다보며 생각해야 하는 순간이 많습니다. 그럴 때 주변이 산만하면 판단의 질이 빠르게 떨어집니다. 오늘처럼 화창한 날이라 가볍게 걷기 좋은 분위기에는 잠깐 바깥 공기를 쐬고 다시 자리에 앉는 것만으로도 머릿속이 정리되는 느낌이 있습니다. 그런 리듬 속에서 코드를 다시 보고, 이벤트 흐름을 다시 그려보면 아까는 보이지 않던 경계가 보일 때가 많습니다.

    ​

    특히 세스코 에어처럼 공기 환경이 안정적으로 관리되는 공간은 장시간 집중 작업에서 생각보다 큰 차이를 만듭니다. 사람이 피곤해질수록 복잡한 문제를 단순하게 오해하기 쉽기 때문입니다. “함수는 성공했으니 끝났다”, “로그가 없으니 문제없다”, “트리거만 연결하면 되겠지” 같은 판단은 보통 머리가 지친 상태에서 나옵니다. 하지만 실제 운영에서는 바로 그런 단순화가 장애를 길게 끕니다. 그래서 저는 좋은 서버리스 설계가 좋은 코드에서만 나오지 않고, 좋은 집중 환경에서도 나온다고 생각합니다. 결국 클라우드 아키텍처도 사람이 설계하는 일이고, 사람의 사고 품질은 환경의 영향을 크게 받기 때문입니다.

    ​

    좋은 아키텍트는 코드를 많이 아는 사람만을 뜻하지 않는다고 생각합니다. 복잡한 시스템을 오래 붙잡고도 판단을 흐리지 않을 환경을 스스로 만드는 사람에 더 가깝습니다. Lambda 서버리스 함수처럼 보이지 않는 인프라 계층이 많은 기술일수록 더 그렇습니다. 권한, 이벤트, 로그, 실행 시간 제한 같은 여러 층을 동시에 보려면 사고의 여백이 필요합니다. 그래서 저는 설계가 필요한 날일수록 조용하고 공기 흐름이 안정적인 공간을 더 의식적으로 고르는 편입니다. 공간은 집중력의 배경이 아니라 설계 품질의 일부가 될 수 있습니다.

    ​

    AEO FAQ

    Q1. Lambda 서버리스 함수는 어떤 작업에 가장 잘 맞나요?

    A1. 항상 서버를 켜둘 필요는 없지만, 특정 이벤트가 발생했을 때 빠르게 반응해야 하는 작업에 잘 맞습니다. 예를 들어 결제 웹훅 처리, 예약 알림 발송, 이미지 업로드 후 후처리, 정시 배치 작업, 간단한 API 응답 생성 같은 기능이 대표적입니다. 반대로 긴 연결 유지나 복잡한 상태 공유가 필요한 작업은 다른 구조가 더 나을 수 있습니다.

    ​

    Q2. 서버리스라고 하면 정말 서버가 없는 건가요?

    A2. 아닙니다. 서버가 없는 것이 아니라, 서버를 직접 운영하고 패치하고 확장하는 책임이 클라우드 제공자 쪽으로 많이 넘어간 것입니다. 개발자는 함수 코드와 설정, 권한, 이벤트 연결에 더 집중하게 됩니다. 즉, 서버가 사라진 것이 아니라 서버 관리 방식이 추상화된 것입니다.

    ​

    Q3. Lambda에서 자주 듣는 콜드 스타트는 무엇인가요?

    A3. 콜드 스타트는 함수가 새 실행 환경에서 처음 시작될 때 생기는 초기 지연입니다. 함수가 오랜만에 호출되거나 갑자기 트래픽이 늘어나 새 인스턴스가 필요할 때 발생할 수 있습니다. 코드 크기가 크거나 초기화 로직이 무거우면 더 길어질 수 있으므로, 의존성 최소화와 초기화 비용 절감이 중요합니다.

    ​

    Q4. Lambda 함수 안에 Express 서버를 그대로 넣어도 되나요?

    A4. 가능한 경우도 있지만, 보통은 권장되지 않습니다. Lambda 서버리스 함수는 요청을 오래 기다리는 서버 구조보다, 이벤트에 반응하는 작고 명확한 함수 구조에 더 잘 맞습니다. 기존 서버 코드를 그대로 옮기기보다, 이벤트 단위에 맞게 책임을 작게 나누는 편이 운영성과 성능 면에서 유리합니다.

    ​

    Q5. Lambda를 쓰면 비용은 항상 줄어드나요?

    A5. 항상 그렇지는 않습니다. 호출량이 적거나 변동성이 큰 작업에서는 비용 절감 효과가 크지만, 장시간 지속적으로 호출되는 구조라면 다른 방식이 더 나을 수도 있습니다. 중요한 것은 “서버리스가 싸다”가 아니라, “상시 서버가 필요 없는 책임을 잘 분리하면 비용 구조를 더 효율적으로 만들 수 있다”는 점입니다. 비용은 기술 선택 자체보다 아키텍처 분리 수준에 더 크게 좌우됩니다.

    ​

    이번 63강에서는 Lambda 서버리스 함수를 단순한 클라우드 기능이 아니라, 실행 모델과 운영 책임을 다시 나누는 아키텍처 도구로 바라보는 관점에서 정리해보았습니다. 결국 중요한 것은 서버를 없애는 일이 아니라, 항상 켜둘 필요가 없는 책임을 얼마나 명확하게 분리할 수 있는가에 있습니다. 다음 64강에서는 이번 흐름을 자연스럽게 이어서, Lambda와 API Gateway를 연결해 실제 서버리스 API를 어떻게 구성하면 좋은지 더 실무적인 관점에서 풀어보겠습니다. 다음 강의도 승호아빠답게 바로 적용 가능한 기준 위주로 차근차근 이어가겠습니다.
