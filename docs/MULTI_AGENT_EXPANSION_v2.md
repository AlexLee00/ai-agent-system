## 1. 비전

```
현재 (v1): 고정 편성
  제이(메인봇) → 팀장 → 팀원 (항상 같은 조합)

v2 목표: 동적 고용 + 자기학습
  제이(메인봇) → 팀장(목표 달성) → 에이전트 풀(동적 고용/해고)
  팀장이 목표에 맞는 최적 에이전트를 풀에서 선택하여 작업 수행

v3 목표: 그룹 경쟁 + 최적화
  팀장 아래 여러 그룹이 동일 작업을 병렬 수행
  결과물 비교 → 최선 채택 → 패턴 RAG 저장 → 자동 최적화
```

## 2. 핵심 아이디어 (마스터 구상)

### 2-1. 다층 구조 확장
```
제이(메인봇) → 팀장(역할에 목표달성) → 여러 에이전트(동일/다른 역할 다수)

예시 — 블로팀:
  블로(팀장, 목표: "IT 강의 포스팅")
    → 풀에서 고용: IT기획자 + 기술작가 + IT수집 + SEO편집
  
  블로(팀장, 목표: "감성 자기계발 포스팅")  
    → 풀에서 고용: 감성기획자 + 에세이작가 + 심리학수집 + 문체편집

  같은 "작가" 역할이지만 IT작가와 감성작가는 다른 에이전트!
```

### 2-2. 에이전트 풀 (Agent Registry) — 마스터 확정
```
기존 에이전트를 기반으로 확장 (완전 새로 만들지 않음)
연구팀이 연구한 내용을 에이전트에 적용

확장 유형:
  A. 역할 확장: 기존과 다른 역할의 에이전트 추가
     예) 블로팀에 "SEO분석가" 역할 신설
  B. 성향 분화: 동일 역할 + 다른 성향
     예) IT기술작가(논리적) vs 감성에세이작가(감성적)
  C. 연구팀 적용: 연구팀이 발견한 최적 패턴을 에이전트 프롬프트에 반영
     예) "3단 구조가 9,000자에 최적" → 작가 에이전트에 적용

에이전트는 역할(Role) + 전문분야(Specialty) + 성과점수(Score)로 관리
팀장이 목표를 받으면:
  1) 필요한 역할 목록 산출
  2) 각 역할에서 성과점수 높은 에이전트 우선 고용
  3) 점수 낮은 에이전트 → 학습 강화 프로그램으로 보완
```

### 2-3. 그룹 경쟁 (v3) — 마스터 확정
```
팀별 차등 적용:

블로팀: 최대 2개 그룹 병행
  → 그룹A vs 그룹B 병렬 실행
  → 결과물 품질 비교 (LLM 심사 또는 규칙 기반)
  → 승리 그룹 패턴 → RAG 저장
  → 비용 2배지만, 콘텐츠 품질 = 직접 수익 연결

루나팀: 병행 없이 팀 구성만 변경
  → 기존 에이전트 역할/가중치 조정
  → 전략별 최적 에이전트 조합 탐색
  → 경쟁 아닌 최적화 (실투자 안정성 우선)
```

### 2-4. N8N 워크플로우 — 마스터 확정: 도입
```
N8N 도입 확정 (기존 "불필요" → "도입")
업무에 대한 워크플로우 템플릿 제공
  수집 → 기획 → 작성 → 편집 → 검증 → 발행
에이전트는 워크플로우의 각 단계를 담당
N8N이 순서/병렬/조건 분기를 관리
결정적 파이프라인 (LLM이 흐름 결정 안 함)
```

### 2-5. RAG 전중후 피드백
```
전(Before): 과거 성공 편성 조합 검색
중(During): 실행 중간 결과 품질 점수 저장
후(After): 최종 결과 + 성과 평가 triplet 저장
→ 다음번 같은 유형 → 최적 조합 자동 추천
```

### 2-6. 모니터링 대시보드 — 마스터 확정: 워커 포털에 추가
```
워커 포털(port 4001)에 "에이전트 오피스" 페이지 추가
QJC OS Visualizer 스타일 — 팀별 에이전트 카드 뷰
  각 팀 = 색상 구분 카드 그룹 (파랑/초록/보라/갈색 등)
  개별 에이전트 = 카드 형식:
    - 이름 + 아이콘
    - 역할 + 전문분야
    - 성과 점수 (시각적 표시)
    - 현재 상태 (활성/대기/학습중)
    - 최근 작업 이력 요약
  상단: 전체 통계 (활성 에이전트 수, 작업 진행 수, 성과 트렌드)

향후 규모 확장 시: 별도 포트 또는 독립 대시보드로 분리 가능
```

### 2-7. 학습 강화 프로그램 (신규)
```
점수 낮은 에이전트 → 자동 학습 강화:
  1) 실패 패턴 분석: corrections.md에서 반복 실패 추출
  2) 성공 사례 학습: RAG에서 같은 역할의 성공 경험 주입
  3) 프롬프트 튜닝: 에이전트 시스템 프롬프트 자동 보정
  4) 재시험: 동일 작업 재실행 → 점수 재평가
  5) 복귀 또는 퇴역: 개선 시 풀 복귀, 미개선 시 아카이브
```

## 3. 커뮤니티 리서치 결과

### 3-1. CrewAI — 역할 기반 팀 편성 (가장 유사)
```
패턴: Agent(role, goal, backstory) + Task + Crew
  - Hierarchical Process: 매니저가 태스크 할당+결과 검토+동적 위임
  - Sequential Process: 순서대로 실행, 이전 결과를 다음에 전달
  - 커뮤니티 요청: "Dynamic Agent Selection" — 쿼리에 따라 관련 에이전트만 동적 선택
    (제이 아이디어와 정확히 동일한 고민, 아직 공식 해답 없음)
  - 팀 제이 차별화: CrewAI는 고정 Crew 편성, 우리는 성과 기반 동적 고용
```

### 3-2. OpenClaw Sub-Agents — 이미 보유한 인프라
```
sessions_spawn: 백그라운드 에이전트 스폰 (격리 세션)
  - 비차단, 완료 시 결과 announce back
  - 에이전트별 다른 모델 설정 가능 (비용 절감)
  - maxSpawnDepth=2: 오케스트레이터→워커 패턴 지원
  - maxChildrenPerAgent=5: 동시 5개 서브에이전트
  
sessions_send: 피어 에이전트 간 직접 메시지
  - 서브에이전트가 아닌 독립 에이전트 간 통신
  - allowlist로 통신 범위 제한
```

### 3-3. Lobster — 결정적 워크플로우 엔진
```
한 개발자의 교훈:
  "LLM이 흐름을 결정하면 비결정적 → 예측 불가"
  → 워크플로우(순서)는 코드로, 창작은 LLM으로 분리
  → Lobster(OpenClaw 워크플로우 엔진)에 루프 지원 기여
  
팀 제이 적용:
  - 워크플로우 = N8N 또는 자체 Task Orchestrator (결정적)
  - 콘텐츠 = LLM 에이전트 (비결정적, 창작)
```

### 3-4. Medium — "Single Agent to Team Architecture" 전환기
```
문제: 단일 에이전트가 모든 일 → 메모리 컨텍스트 누출 + 프롬프트 40K 토큰
해결: 에이전트별 별도 워크스페이스 + 메모리 격리
→ 팀 제이: 이미 팀별 분리 완료, v2에서 에이전트별 분리 추가 필요
```

### 3-5. 커뮤니티에 없는 것 (팀 제이 차별화)
```
1. Agent Registry + 성과 랭킹 → 아무도 안 만듦
2. Competition Engine (그룹 A vs B) → 패턴 없음
3. Dynamic Hiring Logic (목표→역할분해→최적선택) → CrewAI 포럼 질문만 있음
4. 학습 강화 프로그램 (저성과 에이전트 재교육) → 없음
5. 에이전트 모니터링 대시보드 (카드 뷰 + 성과) → 없음
```

## 5. 구현 로드맵

### Phase 0: 현재 진행 중 (즉시~1주)
```
이미 진행 중인 작업 완료 우선:
  ✅ LLM 모델 재편성 (코덱스 구현 완료, 수정 2건 남음)
  ⏳ Phase 4 alert resolve OpenClaw 통합 (코덱스 진행중, 검증 대기)
  ✅ self-improving 스킬 설치 + 셋업
  📋 RAG 경험 저장 (CODEX_SELF_IMPROVING_RAG.md 프롬프트 완료)
  📋 블로팀 P1~P5 개선
```

### Phase 1: 기반 구축 (2~4주) — 블로팀 + 루나팀 동시 착수
```
1-1. RAG 경험 저장 구현
  → rag_experience 컬렉션 + experience-store.js + CLI
  → 성공/실패 triplet 누적 시작

1-2. Agent Registry 설계
  → agent_registry 테이블 (PostgreSQL)
  → 컬럼: id, name, role, specialty, llm_model, score, status, config
  → 초기 데이터: 기존 56 에이전트 등록

1-3. 블로팀 에이전트 세분화 (첫 적용 대상 ①)
  → 작가: IT기술작가, 감성에세이작가, 분석리포트작가, 교육튜토리얼작가
  → 기획: IT커리큘럼기획, 트렌드기획, 감성주제기획
  → 수집: IT뉴스수집, 도서정보수집, 심리학수집, SEO분석수집
  → 편집: 문체통일편집, SEO최적화편집, 이미지선정편집

1-4. 루나팀 에이전트 재편성 (첫 적용 대상 ②)
  → 기존 11 에이전트 역할/가중치 재조정
  → 전략별 최적 조합 탐색 (그룹 경쟁 없이 구성만 변경)
  → 연구팀 연구 결과 적용 (Chronos Layer 2~3 등)

1-5. 성과 평가 시스템
  → 작업 완료 시 점수 기록 (품질, 속도, 비용)
  → 에이전트별 평균 점수 + 트렌드
```

### Phase 2: 동적 고용 + N8N (1~2개월)
```
2-1. Task Orchestrator
  → 팀장이 목표 분석 → 필요 역할 산출
  → Agent Registry에서 최적 에이전트 선택
  → 워크플로우 편성 (순차/병렬)

2-2. 모니터링 대시보드 v1
  → 워커 포털(port 4001)에 에이전트 오피스 페이지 추가
  → 팀별 카드 그룹 + 개별 에이전트 카드
  → 실시간 상태 + 성과 점수 + 최근 작업

2-3. N8N 워크플로우 도입 (마스터 확정)
  → 블로팀 워크플로우 템플릿: 수집→기획→작성→편집→검증
  → 루나팀 워크플로우: 수집→분석→토론→판단→실행
  → 에이전트 간 데이터 전달 파이프라인
  → 실패 시 대체 에이전트 교체 로직
```

### Phase 3: 그룹 경쟁 + 학습 강화 (2~3개월)
```
3-1. Competition Engine (블로팀 전용, 최대 2그룹)
  → 그룹A vs 그룹B 병렬 실행 (블로팀만)
  → 품질 평가 엔진 (LLM 심사 + 규칙 기반)
  → 승리 패턴 RAG 저장
  → 루나팀은 경쟁 없이 구성 최적화만

3-2. 학습 강화 프로그램
  → 저성과 에이전트 자동 감지 (점수 하위 20%)
  → 실패 패턴 분석 + 성공 사례 주입
  → 프롬프트 자동 튜닝 + 재시험
  → 복귀 또는 아카이브 판단

3-3. 모니터링 대시보드 v2
  → 그룹 경쟁 결과 시각화
  → 학습 강화 진행 상태
  → 에이전트 성과 트렌드 차트

3-4. 전체 팀 적용 확대
  → 블로팀+루나팀에서 검증된 패턴을 나머지 팀에 적용
  → 9팀 순서: 블로→루나→스카→클로드→워커→에디→연구→감정→제이
```

## 7. 핵심 원칙

```
1. 점진적 확장: 블로팀에서 먼저 검증 → 성공 시 확대
2. 비용 의식: 서브에이전트는 저렴한 모델, 경쟁은 중요 작업에만
3. 데이터 기반: 모든 결정은 RAG 경험 + 성과 점수 기반
4. 자율+통제: 에이전트는 자율 실행, 팀장이 통제, 마스터가 승인
5. 기록이 곧 진화: 모든 작업의 전중후를 기록 → 학습 → 개선
6. 메모리 격리: 에이전트별 워크스페이스 분리 → 컨텍스트 누출 방지
7. 프레임워크 독립: CrewAI/LangGraph 패턴 참조, 직접 설치 안 함
```
