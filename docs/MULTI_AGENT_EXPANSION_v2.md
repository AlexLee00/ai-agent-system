## 1. 비전

```
현재 (v1): 고정 편성
  제이(메인봇) → 팀장 → 팀원 (항상 같은 조합)

v2 목표: 동적 고용 + 자기학습
  제이(메인봇) → 팀장(목표 달성) → 에이전트 풀(동적 고용/해고)
  팀장이 목표에 맞는 최적 에이전트를 풀에서 선택하여 작업 수행

v3 목표: 그룹 경쟁 + 최적화
  팀장 아래 여러 그룹이 동일 작업을 병렬 수행
  결과물 비교 → 최선 채택 → 패턴 RAG 저장 → 자동 최적화
```

## 2. 핵심 아이디어 (마스터 구상)

### 2-1. 다층 구조 확장
```
제이(메인봇) → 팀장(역할에 목표달성) → 여러 에이전트(동일/다른 역할 다수)

예시 — 블로팀:
  블로(팀장, 목표: "IT 강의 포스팅")
    → 풀에서 고용: IT기획자 + 기술작가 + IT수집 + SEO편집
  
  블로(팀장, 목표: "감성 자기계발 포스팅")  
    → 풀에서 고용: 감성기획자 + 에세이작가 + 심리학수집 + 문체편집

  같은 "작가" 역할이지만 IT작가와 감성작가는 다른 에이전트!
```

### 2-2. 에이전트 풀 (Agent Registry) — 마스터 확정
```
기존 에이전트를 기반으로 확장 (완전 새로 만들지 않음)
연구팀이 연구한 내용을 에이전트에 적용

확장 유형:
  A. 역할 확장: 기존과 다른 역할의 에이전트 추가
     예) 블로팀에 "SEO분석가" 역할 신설
  B. 성향 분화: 동일 역할 + 다른 성향
     예) IT기술작가(논리적) vs 감성에세이작가(감성적)
  C. 연구팀 적용: 연구팀이 발견한 최적 패턴을 에이전트 프롬프트에 반영
     예) "3단 구조가 9,000자에 최적" → 작가 에이전트에 적용

연구팀 → 에이전트 진화 사이클:
  1) 연구팀이 관련 기술/논문/커뮤니티 서칭
  2) 코드로 구현 + 실험 + 연구
  3) 연구 결과를 새 에이전트로 생성 또는 기존 에이전트에 기술 반영
  4) 필드(실제 업무)에 투입
  5) 피드백 수집 (RAG 경험 저장)
  6) 피드백 기반 다음 연구 사이클

  → 동일한 작업을 수행하더라도 에이전트마다 기능/성향 차이 발생
  → 이 차이가 그룹 경쟁의 다양성을 만들어냄

저성과 에이전트 심층 분석 프로세스:
  1) 성과 점수 하위 20% 자동 감지
  2) 연구팀이 심층 분석 (실패 원인, 프롬프트 문제, 모델 한계 등)
  3) 개선 방안 연구 + 프롬프트/모델/파이프라인 수정
  4) 개선 결과를 해당 에이전트뿐 아니라 전체 반영 검토
  5) 전체 에이전트에 공통 개선사항 적용 → 시스템 전체 수준 상승

에이전트는 역할(Role) + 전문분야(Specialty) + 성과점수(Score)로 관리
팀장이 목표를 받으면:
  1) 필요한 역할 목록 산출
  2) 각 역할에서 성과점수 높은 에이전트 우선 고용
  3) 점수 낮은 에이전트 → 학습 강화 프로그램으로 보완
```

### 2-3. 그룹 경쟁 (v3) — 마스터 확정
```
팀별 차등 적용:

블로팀: 최대 2개 그룹 병행
  → 그룹A vs 그룹B 병렬 실행
  → 결과물 품질 비교 (LLM 심사 또는 규칙 기반)
  → 승리 그룹 패턴 → RAG 저장
  → 비용 2배지만, 콘텐츠 품질 = 직접 수익 연결

루나팀: 병행 없이 팀 구성만 변경
  → 기존 에이전트 역할/가중치 조정
  → 전략별 최적 에이전트 조합 탐색
  → 경쟁 아닌 최적화 (실투자 안정성 우선)
```

### 2-4. N8N 워크플로우 — 마스터 확정: 도입
```
N8N 도입 확정 (기존 "불필요" → "도입")
업무에 대한 워크플로우 템플릿 제공
  수집 → 기획 → 작성 → 편집 → 검증 → 발행
에이전트는 워크플로우의 각 단계를 담당
N8N이 순서/병렬/조건 분기를 관리
결정적 파이프라인 (LLM이 흐름 결정 안 함)
```

### 2-5. RAG 전중후 피드백
```
전(Before): 과거 성공 편성 조합 검색
중(During): 실행 중간 결과 품질 점수 저장
후(After): 최종 결과 + 성과 평가 triplet 저장
→ 다음번 같은 유형 → 최적 조합 자동 추천
```

### 2-6. 모니터링 대시보드 — 마스터 확정: 워커 포털에 추가
```
워커 포털(port 4001)에 "에이전트 오피스" 페이지 추가
QJC OS Visualizer 스타일 — 팀별 에이전트 카드 뷰

상단 고정: 상시 가동 에이전트 상태 바 (마스터 아이디어)
  → 24/7 상시 가동 에이전트의 실시간 작동 상태를 상단에 고정 표시
  → 덱스터(시스템 감시), 앤디(예약 감지), 이브(환경 수집), 허브(API) 등
  → 상태: 🟢 정상 / 🟡 지연 / 🔴 중단 / ⚪ 비활성
  → 마지막 활동 시각 + 헬스 점수
  → 문제 발생 시 즉시 시각적 인지 가능

에이전트 도트 캐릭터 (마스터 아이디어):
  → 각 에이전트마다 특색있는 도트(픽셀) 캐릭터!
  → 역할과 성격을 반영한 고유한 디자인
  → 예시:
    포스(IT작가): 안경 쓴 개발자 도트 캐릭터 🤓
    젬스(감성작가): 펜 들고 있는 작가 도트 캐릭터 ✍️
    루나(투자팀장): 냉철한 분석가 도트 캐릭터 🧊
    덱스터(시스템감시): 돋보기 든 탐정 도트 캐릭터 🔍
    아리아(기술분석): 차트 든 분석가 도트 캐릭터 📊
    네메시스(리스크): 방패 든 수호자 도트 캐릭터 🛡️
  → 상태에 따라 캐릭터 표정/색상 변화
    활성: 밝은 색 + 활기찬 표정
    대기: 흐린 색 + 졸린 표정
    학습중: 책 읽는 포즈
    에러: 빨간 깜빡임

  각 팀 = 색상 구분 카드 그룹 (파랑/초록/보라/갈색 등)
  개별 에이전트 = 카드 형식:
    - 이름 + 아이콘
    - 역할 + 전문분야
    - 성과 점수 (시각적 표시)
    - 현재 상태 (활성/대기/학습중)
    - 최근 작업 이력 요약
  상단: 전체 통계 (활성 에이전트 수, 작업 진행 수, 성과 트렌드)

동적 렌더링 — 계약 기반 그룹 구성 변화 (마스터 아이디어):
  에이전트가 고용 계약을 맺으면 → 해당 팀 그룹에 카드가 추가
  계약 종료/해고 → 카드가 "에이전트 풀"로 이동
  그룹 경쟁 시 → 그룹A/그룹B 카드가 동적으로 분리 표시
  학습 강화 편입 → 카드가 "학습중" 상태로 변경 (색상/뱃지)

  실시간 시나리오 예시:
    14:00 — 블로(팀장)이 "IT 강의 포스팅" 목표 수신
    14:01 → IT기획자A 카드가 풀→블로팀 그룹A로 이동 (고용 애니메이션)
    14:01 → IT작가B 카드가 풀→블로팀 그룹A로 이동
    14:01 → 감성기획자C 카드가 풀→블로팀 그룹B로 이동 (경쟁 그룹)
    14:01 → 감성작가D 카드가 풀→블로팀 그룹B로 이동
    14:30 → 그룹A 결과 품질 9.2 / 그룹B 결과 품질 8.5
    14:31 → 그룹A 승리 표시 (카드 하이라이트)
    14:32 → 계약 종료 → 모든 카드가 풀로 복귀
    마스터는 이 전체 과정을 워커 웹에서 실시간으로 관찰!

  UI 구현:
    에이전트 풀: 화면 좌측 사이드바 (대기 중 에이전트 목록)
    팀 작업 영역: 중앙 (팀별 그룹, 그룹A/B 분리 가능)
    학습 강화: 우측 (학습 중인 에이전트)
    카드 이동: CSS transition/animation (드래그앤드롭 느낌)
    상태 변화: WebSocket 실시간 업데이트

향후 규모 확장 시: 별도 포트 또는 독립 대시보드로 분리 가능
```

### 2-7. 학습 강화 프로그램 (신규)
```
점수 낮은 에이전트 → 자동 학습 강화:
  1) 실패 패턴 분석: corrections.md에서 반복 실패 추출
  2) 성공 사례 학습: RAG에서 같은 역할의 성공 경험 주입
  3) 프롬프트 튜닝: 에이전트 시스템 프롬프트 자동 보정
  4) 재시험: 동일 작업 재실행 → 점수 재평가
  5) 복귀 또는 퇴역: 개선 시 풀 복귀, 미개선 시 아카이브
```

### 2-8. 고용 계약 시스템 (신규) — 마스터 아이디어
```
에이전트는 "고용 계약"을 통해 작업에 투입된다.
계약이 에이전트의 최선을 이끌어내는 구조적 장치.

계약서 구조:
  contract_id   — 고유 계약 ID
  employer      — 팀장 (고용주)
  agent         — 에이전트 (피고용자)
  task          — 작업 내용 + 요구사항 (글자수, 섹션, 품질 기준)
  reward        — 보상 (기본 점수 + 품질 보너스 + 속도 보너스)
  penalty       — 페널티 (품질 미달, 시간 초과, hallucination)
  terms         — 조건 (최대 재시도, 교체 가능 여부)

에이전트가 최선을 다하게 하는 5가지 메커니즘:

  ① 성과 기반 고용 (Performance-based Hiring)
     점수 높은 에이전트가 먼저 고용됨
     점수 낮으면 고용 안 됨 = 다음 기회 상실
     → "일을 잘 해야 다음에도 고용된다"

  ② 자가 확신도 보고 (Confidence Scoring)
     에이전트가 결과물에 확신도(1~10)를 자가 보고
     확신 높고 결과 좋음 = 높은 보상
     확신 높고 결과 나쁨 = 높은 감점 (허세 페널티)
     → 정직한 자기 평가를 유도

  ③ 경쟁 압력 (Competition Pressure)
     같은 역할의 에이전트가 여러 명 = 대체 가능
     그룹 경쟁에서 지면 다음 고용 확률 감소
     → "나보다 잘하는 에이전트가 있다"

  ④ 하이브리드 인센티브 (Hybrid Incentives)
     보상만: 위험 행동 (빠르게만, 품질 무시)
     페널티만: 소극적 행동 (안전한 것만)
     하이브리드: 품질 보너스 + 실패 감점 = 균형
     hallucination 페널티 최고 (-1.0)

  ⑤ 학습 강화 복귀 기회 (Rehabilitation Path)
     저성과 = 바로 퇴역이 아닌 연구팀 심층 분석 → 개선
     개선 후 재시험 → 통과 시 풀 복귀
     → "노력하면 복귀할 수 있다"

학술 근거:
  - Principal-Agent Contract Theory (2024 노벨경제학상 기반, arXiv 2407.18074)
  - Confidence-weighted Rewards (arXiv 2505.18286)
  - Auto-adjusting Incentives — 보상 22% 향상 (arXiv 2409.02960)
  - 안전 주의: 인센티브 과도 시 윤리/안전 무시 가능 (arXiv 2512.20798)

LLM 에이전트에서의 구현:
  A. 프롬프트 엔지니어링: 계약서를 시스템 프롬프트에 포함
  B. 모델 차등 배정: 중요 작업 = 고성능 모델
  C. 품질 게이트: 미달 시 자동 재시도 → 2회 실패 시 교체
  D. RAG 주입: 과거 성공 패턴을 프롬프트에 포함
  E. 점수 → Registry 반영: 자동 루프로 다음 고용에 영향
```

### 2-9. 민원게시판 (Agent Feedback Board) — 마스터 아이디어
```
에이전트가 작업 중 발견한 문제점, 개선 요청, 건의사항을 등록하는 공식 채널.
에이전트도 팀 제이의 구성원으로서 목소리를 낼 수 있는 구조.

핵심 원칙: 완전한 독립적 자유
  → 민원 등록부터 해결까지 에이전트들이 자율적으로 처리
  → 마스터는 워커 웹에서 결과만 확인
  → 사람이 개입하지 않아도 돌아가는 자율 개선 시스템

운영 흐름:
  1) 에이전트가 작업 중 문제 발견 → 민원게시판에 티켓 등록
  2) 담당 팀에 자동 배정 (역할 기반 라우팅)
  3) 담당 팀이 요청사항 확인 → 분석 → 개선 작업 수행
  4) 결과보고를 다시 게시판에 올림 (해결 내역 + 적용 범위)
  5) 원래 요청자가 결과 확인 → 완료 또는 재요청
  6) 반복 민원 3건+ → 시스템 개선 과제로 자동 승격

등록 가능 유형:
  🔧 버그 리포트: "이 API가 404 반환됨" "DB 타임아웃 발생"
  💡 개선 제안: "프롬프트에 X 추가하면 품질 올라갈 것" "이 워크플로우 비효율적"

## 3. 커뮤니티 리서치 결과

### 3-1. CrewAI — 역할 기반 팀 편성 (가장 유사)
```
패턴: Agent(role, goal, backstory) + Task + Crew
  - Hierarchical Process: 매니저가 태스크 할당+결과 검토+동적 위임
  - Sequential Process: 순서대로 실행, 이전 결과를 다음에 전달
  - 커뮤니티 요청: "Dynamic Agent Selection" — 쿼리에 따라 관련 에이전트만 동적 선택
    (제이 아이디어와 정확히 동일한 고민, 아직 공식 해답 없음)
  - 팀 제이 차별화: CrewAI는 고정 Crew 편성, 우리는 성과 기반 동적 고용
```

### 3-2. OpenClaw Sub-Agents — 이미 보유한 인프라
```
sessions_spawn: 백그라운드 에이전트 스폰 (격리 세션)
  - 비차단, 완료 시 결과 announce back
  - 에이전트별 다른 모델 설정 가능 (비용 절감)
  - maxSpawnDepth=2: 오케스트레이터→워커 패턴 지원
  - maxChildrenPerAgent=5: 동시 5개 서브에이전트
  
sessions_send: 피어 에이전트 간 직접 메시지
  - 서브에이전트가 아닌 독립 에이전트 간 통신
  - allowlist로 통신 범위 제한
```

### 3-3. Lobster — 결정적 워크플로우 엔진
```
한 개발자의 교훈:
  "LLM이 흐름을 결정하면 비결정적 → 예측 불가"
  → 워크플로우(순서)는 코드로, 창작은 LLM으로 분리
  → Lobster(OpenClaw 워크플로우 엔진)에 루프 지원 기여
  
팀 제이 적용:
  - 워크플로우 = N8N 또는 자체 Task Orchestrator (결정적)
  - 콘텐츠 = LLM 에이전트 (비결정적, 창작)
```

### 3-4. Medium — "Single Agent to Team Architecture" 전환기
```
문제: 단일 에이전트가 모든 일 → 메모리 컨텍스트 누출 + 프롬프트 40K 토큰
해결: 에이전트별 별도 워크스페이스 + 메모리 격리
→ 팀 제이: 이미 팀별 분리 완료, v2에서 에이전트별 분리 추가 필요
```

### 3-5. 커뮤니티에 없는 것 (팀 제이 차별화)
```
1. Agent Registry + 성과 랭킹 → 아무도 안 만듦

4. 학습 강화 프로그램 (저성과 에이전트 재교육) → 없음
5. 에이전트 모니터링 대시보드 (카드 뷰 + 성과) → 없음
```

### 데이터 사이언스 팀 (신설) — 마스터 아이디어
```
현재: 아처(기술 인텔리전스), 루나팀(시장 데이터) 등이 이미 정기 수집 중
문제: 수집된 데이터를 체계적으로 분석하고 의미있는 인사이트를 뽑아내는 전담이 없음
해결: 데이터 사이언스 팀 신설

역할:
  1) 정기 데이터 수집 총괄
     → 각 팀이 수집하는 데이터를 중앙에서 관리
     → 수집 스케줄 관리 (매시간/매일/매주)
     → 데이터 품질 검증 (결측, 중복, 이상치)

  2) 데이터 가공 + 분석
     → 원시 데이터 → 정제 → 피처 추출 → 패턴 발견
     → 팀별 성과 상관관계 분석
     → 에이전트 성과 예측 모델
     → 트렌드 분석 + 이상 감지

  3) 의미있는 데이터 아카이빙
     → 인사이트를 RAG에 구조화 저장
     → 의미 없는 데이터 정리 (노이즈 제거)
     → 데이터 카탈로그 관리 (무슨 데이터가 어디에 있는지)

  4) 다른 팀에 데이터 제공
     → 블로팀: 인기 포스팅 패턴 + SEO 트렌드 분석
     → 루나팀: 시장 데이터 패턴 + 전략 백테스트 데이터
     → 연구팀: 에이전트 성과 데이터 + 실험 결과 분석
     → 감정팀: 감정 유형별 정확도 데이터 + 개선 포인트

이미 수집 중인 데이터 (활용 가능):
  → llm_usage_log: 35,000+/주 LLM 호출 기록
  → rag_experience: 에이전트 경험 triplet
  → trade_journal/rationale/review: 매매 기록
  → blog.posts: 포스팅 본문 + 성과
  → agent_events/tasks: 에이전트 활동 이력
  → 아처 수집: GitHub/npm/커뮤니티 기술 동향
  → 루나팀 수집: 바이낸스/업비트/야후 시장 데이터
  → 덱스터 점검: 22개 카테고리 시스템 상태

데이터 사이언스 팀 구성 (초기):
  팀장: 데이터(가칭) — 데이터 총괄, 분석 전략 수립
  수집 에이전트: 각 팀 수집 데이터 통합 파이프라인
  분석 에이전트: 통계 분석 + 패턴 발견 + 시각화
  인사이트 에이전트: 분석 결과 → RAG 저장 + 리포트 생성
```

데이터 3대 질문:
  1) 어떻게 모여있는가? → RAG + PostgreSQL + 경험 저장
  2) 어떻게 가공할 수 있는가? → 분석 + 패턴 추출 + 연구팀 연구
  3) 어떻게 사용할 수 있는가? → 에이전트 투입 + 의사결정 + 수익화

데이터 기반 자율 진화 루프:
  ┌────────────────────────────────────────────────┐
  │  데이터 축적                                     │
  │  (RAG, 경험 저장, 민원, 성과, 계약 이력)           │

                ↓
  ┌────────────────────────────────────────────────┐
  │  자동 분석                                       │
  │  (패턴 추출, 성공/실패 요인, 트렌드, 상관관계)       │
  └─────────────┬──────────────────────────────────┘
                ↓
  ┌────────────────────────────────────────────────┐
  │  자동 설계                                       │
  │  (연구팀: 새 에이전트 설계, 프롬프트 최적화, 전략 수정) │
  └─────────────┬──────────────────────────────────┘
                ↓
  ┌────────────────────────────────────────────────┐
  │  자동 구현                                       │

  └─────────────┬──────────────────────────────────┘
                ↓
  ┌────────────────────────────────────────────────┐

  └─────────────┬──────────────────────────────────┘

                └──→ 다시 데이터 축적으로 (무한 루프)

이 루프가 사람 개입 없이 돌아가면 = 제이 랜드는 스스로 진화한다.
```

### 데이터 계층 (Data Lake)
```
Layer 1: 원시 데이터 (Raw)
  → 에이전트 실행 로그, LLM 호출 기록 (35,000+/주)
  → 텔레그램 메시지, 알람 이력
  → 블로그 포스팅 본문 + 조회수
  → 루나팀 매매 기록 + 시장 데이터
  → 감정팀 분석 리포트

Layer 2: 가공 데이터 (Processed)
  → RAG 벡터 임베딩 (rag_* 10개 컬렉션)
  → 경험 저장 triplet (intent-response-result)
  → 에이전트 성과 점수 + 트렌드
  → 민원게시판 패턴 분석

Layer 3: 인사이트 (Insights)
  → 성공 패턴 (어떤 조합이 최적인가)
  → 실패 원인 (왜 품질이 떨어졌는가)
  → 예측 (다음에 무엇이 효과적일까)
  → Standing Orders 자동 승격 (3회 반복 → 규칙화)

Layer 4: 의사결정 + 수익화 (Action)
  → 에이전트 자동 고용/해고
  → 전략 자동 조정
  → 블로그: 인기 패턴 자동 적용 → 조회수 → 광고 수익
  → 루나팀: 성공 전략 자동 가중 → 수익률 → 투자 수익
  → 감정팀: 정확도 향상 → 수임 증가 → 사업 수익
  → 제이 랜드 자체가 머니를 버는 구조
```

> 연구팀이 순차적으로 구현하여 제이 랜드를 더 탄탄하게 만든다.

### 10-1. 에이전트 내적 상태 엔진 (Emotion/Motivation Engine)
```
근거: AgentSociety (arXiv, 2025) — 10,000 에이전트에 감정/욕구/동기 부여

에이전트에게 내적 상태를 부여:
  confidence(자신감): 연속 성공 → 상승, 연속 실패 → 하락
  fatigue(피로도): 연속 작업 → 상승, 휴식 → 하락
  motivation(동기): 고용 빈도 → 상승, 미고용 → 하락

내적 상태가 행동에 영향:
  자신감 높음 → 더 도전적 작업 수행 가능
  피로도 높음 → 학습 강화 프로그램 자동 편입
  동기 낮음 → 연구팀 심층 분석 트리거

구현: 프롬프트에 내적 상태 포함
  "당신은 최근 5건 연속 성공으로 자신감이 높다 (8/10).
   이번 작업은 평소보다 도전적인 수준으로 수행하라."

저장: agent_registry 테이블에 emotion_state JSONB 컬럼
```

### 10-2. Shapley 기여도 측정 (공정한 성과 평가)
```
근거: Shapley-Coop Framework (2025) — 에이전트 한계 기여도 정량화

현재: 개별 품질 평가 (내가 만든 결과물만 평가)
개선: 팀 성과에서 이 에이전트의 실제 기여도 측정

Shapley Value = "이 에이전트가 빠지면 팀 결과가 얼마나 나빠지는가?"

적용:
  그룹 경쟁에서 "누가 진짜 기여했는가" 정확히 측정
  기여도 높은 에이전트 → 더 많은 고용 기회 + 보너스
  기여도 낮은 에이전트 → 역할 재배치 또는 학습 강화

구현: 동일 작업을 에이전트 1명 빼고 실행 → 결과 차이 측정
