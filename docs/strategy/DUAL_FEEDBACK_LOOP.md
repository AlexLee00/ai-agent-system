# 2중 피드백 루프 전략 (Dual Feedback Loop Strategy)

> 작성: 메티 (Claude Opus 4.6)
> 작성일: 2026-04-06
> 버전: v1.0
> 분류: 전략 문서 (STRATEGY 연계)

---

## 1. 핵심 원칙

팀 제이의 피드백 루프는 **2중 구조**로 작동한다.

```
Layer 1: 팀 내부 자체 학습 — 각 팀이 자기 활동 데이터로 자체 개선
Layer 2: 시그마팀 크로스팀 분석 — 전체 팀 데이터 수집→분석→최적화→각 팀 피드백
```

Layer 1만 있으면 각 팀은 개선되지만 전체 최적화가 불가능하다.
Layer 2만 있으면 전체는 보이지만 즉각적 자체 개선이 불가능하다.
**2중 = 개별 학습 + 전체 최적화 동시 달성.**

---

## 2. Layer 1: 팀 내부 자체 학습

각 팀이 자기 업무 결과를 자기 다음 업무에 즉시 반영하는 루프.
**시그마팀 개입 없이** 팀 자체적으로 동작한다.

### 2-1. 루나팀 (투자/매매)

```
사이클: 매매 체결 → 손익 결과 → 에이전트 성과 평가 → 다음 매매에 반영
데이터: trade_rationale, performance_daily, trade_review
피드백 적용:
  → hiring-contract 점수 업데이트 (정확한 분석가 우선 선택)
  → 전략 파라미터 자동 조정 (승률 높은 전략 가중치↑)
  → aria_accurate/sophia_accurate → 다음 매매에서 정확한 에이전트 우선
```

### 2-2. 블로팀 (블로그 자동화)

```
사이클: 글 발행 → 7일 후 조회수/공감 수집 → 작가 성과 평가 → 다음 발행에 반영
데이터: publish_schedule, collect-performance 결과
피드백 적용:
  → hiring-contract 점수 업데이트 (인기 작가 우선 고용)
  → 커리큘럼 주제 자동 조정 (인기 주제 비중↑)
  → 글 품질 기준 자동 보정 (조회수 높은 글의 특성 학습)
```

### 2-3. 다윈팀 (연구/R&D)

```
사이클: 논문 스캔 → 적합성 평가 → 적용 제안 → 승인/거부 → 키워드 진화
데이터: research_papers, proposals, keyword-evolver 결과
피드백 적용:
  → 검색 키워드 자동 진화 (적합률 높은 키워드 가중치↑)
  → 적합성 평가 기준 자동 보정 (승인된 제안의 특성 학습)
  → searcher 점수 업데이트 (적합 논문 많이 찾은 searcher 우선)
```

### 2-4. 클로드팀 (시스템 모니터링)

```
사이클: 에러 감지 → 자동 복구 → 복구 성공/실패 → 복구 패턴 학습
데이터: dexter 점검 결과, doctor 복구 이력, 에러 로그
피드백 적용:
  → 반복 에러 패턴 → Standing Orders 자동 승격
  → 닥터 복구 우선순위 자동 조정
  → 에러 임계값 자동 보정 (오탐/미탐 학습)
```

### 2-5. 스카팀 (스터디카페 관리)

```
사이클: 매출/예약 예측 → 실제 결과 → 예측 오차 분석 → 모델 파라미터 조정
데이터: 매출 데이터, 예약 데이터, 예측 로그
피드백 적용:
  → 예측 모델 파라미터 자동 튜닝
  → 요일/시간대별 패턴 가중치 자동 조정
  → 이상 매출 패턴 감지 시 알림 기준 자동 보정
```

### 2-6. 워커팀 (비즈니스 SaaS)

```
사이클: 태스크 처리 → 처리 시간/품질 → 워크플로우 최적화
데이터: agent_tasks 처리 이력, 사용자 피드백
피드백 적용:
  → 태스크 라우팅 최적화 (빠른 에이전트 우선)
  → 프롬프트 자동 개선 (품질 높은 응답 패턴 학습)
```

### 2-7 ~ 2-9. 저스틴/에디/감정팀

```
동일 패턴: 업무 수행 → 결과 품질 측정 → 에이전트 점수 반영 → 다음 업무에 개선
각 팀의 hiring-contract selectBestAgent가 Layer 1 피드백 엔진 역할
```

## 3. Layer 2: 시그마팀 크로스팀 분석

시그마팀이 **전체 팀의 활동 데이터**를 수집·분석하고,
**각 팀에 피드백**을 제공하는 루프.
Layer 1이 "팀 안의 눈"이라면, Layer 2는 "하늘에서 보는 눈"이다.

### 3-1. 시그마팀 역할 재정의

```
시그마팀 ≠ 데이터 수집팀
시그마팀 = 플랫폼 두뇌 + 지속적 개선 엔진

미션: 10팀 114에이전트의 활동 데이터를 분석하여
      각 팀의 업무 절차를 최적화하고,
      최신 정보를 바탕으로 시스템을 발전시킨다.
```

### 3-2. 시그마팀 3대 축

```
축1: 데이터 수집 + RAG 대도서관 (pipe + curator)
  pipe: 전체 팀 운영 데이터 자동 수집
    → 루나 매매 로그 + 손익 보고서
    → 블로 발행 성과 + 작가별 통계
    → 클로드 에러 로그 + 복구 이력
    → 스카 매출 + 예약 + 예측 오차
    → 다윈 연구 적합률 + 제안 승인율
  curator: pgvector RAG 대도서관 관리
    → intent-response-result triplet 축적
    → 반복 패턴 3회 → Standing Orders 자동 승격
    → 오래된/부정확한 데이터 TTL 정리
    → 크로스팀 지식 공유 (루나 경험 → 블로 활용?)

축2: 분석 + 피드백 루프 (pivot + oracle-ds)
  pivot: 크로스팀 성과 비교 분석
    → 팀별 에이전트 성공률 비교
    → 워크플로우 병목 발견 (어느 파이프라인이 느린가?)
    → 이상 패턴 탐지 (에러 급증? 성과 하락?)
    → 크로스팀 인사이트 (루나 패턴이 블로에도?)
  oracle-ds: 예측 + 최적화 엔진
    → 에이전트 성과 예측 (다음 주 누가 좋을까?)
    → 워크플로우 최적 경로 추천
    → 리소스 할당 최적화 (LLM 호출 비용 분석)

축3: 리포팅 + 워크플로우 개선 (canvas + sigma)
  canvas: 일일/주간/월간 전체 팀 성과 리포트 자동 생성
    → 마스터용 경영 대시보드
    → 텔레그램 일일 요약 발송
  sigma (리더): 전략 수립 + 오케스트레이션
    → 팀 간 우선순위 조정
    → 장기 전략 방향 제시
```

### 3-3. 시그마팀 일일 사이클

```
06:00  pipe → 전체 팀 운영 데이터 수집 → RAG 저장
07:00  pivot → 크로스팀 성과 분석 + 이상 패턴 탐지
08:00  oracle-ds → 예측 모델 실행 + 최적화 제안 생성
09:00  canvas → 일일 리포트 생성 + 텔레그램 발송
지속   curator → RAG 대도서관 관리 + Standing Orders 승격
```

### 3-4. Layer 2 피드백 자동 적용 메커니즘

시그마팀 분석 결과가 각 팀에 **자동으로** 반영되는 4가지 경로:

```
경로 1: hiring-contract 점수 업데이트
  시그마 분석 → 에이전트별 성과 점수 계산 → DB 반영
  → 각 팀 selectBestAgent가 자동으로 개선된 점수 참조
  예: "루나팀 oracle 정확도 78%→65% 하락" → 점수 감점 → 다른 분석가 우선

경로 2: Standing Orders 자동 승격
  시그마 분석 → 반복 패턴 3회 이상 감지 → Standing Orders 등록
  → 해당 팀이 다음부터 자동으로 최적 경로 실행
  예: "클로드팀 MLX 서버 매주 화요일 재시작 필요" → 화요일 자동 재시작 등록

경로 3: 프롬프트/전략 파라미터 자동 튜닝
  시그마 분석 → 최적 파라미터 제안 → config.yaml 또는 DB 반영
  → 각 팀 에이전트가 개선된 파라미터로 실행
  예: "블로팀 강의 포스팅 2000자→2500자가 조회수 32% 높음" → 글자수 기준 자동 조정

경로 4: 워크플로우 병목 자동 제거
  시그마 분석 → 병목 단계 식별 → 워크플로우 경로 변경 제안
  → 마스터 승인 후 적용 (자동 적용 범위는 점진적 확대)
  예: "다윈팀 applicator 372초 중 graft+edison이 100초" → groq 우선 전환 제안
```

## 4. Layer 1 ↔ Layer 2 상호작용

2중 루프는 독립적으로 동작하면서도 서로를 강화한다.

```
Layer 1 → Layer 2 (데이터 공급):
  각 팀의 자체 학습 결과가 시그마팀에 데이터로 전달
  → 자체 학습이 잘 되는 팀 vs 안 되는 팀 비교 가능
  → "루나팀은 자체 학습이 잘 되지만, 블로팀은 피드백 지연이 있다"

Layer 2 → Layer 1 (개선 주입):
  시그마팀 분석 결과가 각 팀의 자체 학습 루프를 개선
  → 자체 학습 기준/파라미터 자체를 최적화
  → "블로팀 피드백 수집 주기를 7일→3일로 단축하면 학습 속도 2배"

시너지 효과:
  Layer 1이 빠른 일상 개선 담당 (즉시 반영)
  Layer 2가 구조적 개선 담당 (패턴 분석 후 반영)
  → 단기 + 장기 개선이 동시에 진행!
```

## 6. 구현 로드맵

```
Phase 1: Layer 1 기반 구축 (현재~)
  ✅ hiring-contract selectBestAgent (블로/루나/다윈)
  ✅ 블로팀 collect-performance (7일 후 조회수 수집)
  ✅ 다윈팀 키워드 진화 (keyword-evolver)
  ✅ 닥터 자율 헬스체크 (scanAndRecover)
  📋 루나팀 매매 결과 → hiring-contract 점수 자동 반영
  📋 클로드팀 에러 패턴 → Standing Orders 자동 승격

Phase 2: Layer 2 시그마팀 데이터 수집
  📋 pipe: 전체 팀 운영 데이터 수집 자동화
  📋 curator: pgvector RAG 대도서관 구축
  📋 수집 데이터 스키마 표준화

Phase 3: Layer 2 분석 + 피드백
  📋 pivot: 크로스팀 성과 분석 자동화
  📋 oracle-ds: 예측 모델 + 최적화 제안
  📋 피드백 4경로 자동 적용 구현

Phase 4: 통합 + 최적화
  📋 canvas: 일일 리포트 자동 생성
  📋 Layer 1↔Layer 2 시너지 최적화
  📋 일일성장 환류 사이클 전팀 완성
```

## 8. 설계 원칙

```
① Layer 1은 즉시, Layer 2는 분석 후
   Layer 1: 업무 완료 직후 점수 반영 (ms 단위)
   Layer 2: 일일 배치 분석 후 반영 (일 단위)

② 자동 적용 범위는 점진적 확대
   초기: hiring-contract 점수 + Standing Orders만 자동
   중기: 프롬프트/파라미터 튜닝 자동
   장기: 워크플로우 구조 변경도 자동 (마스터 승인 후)

③ 모든 피드백은 RAG에 축적
   성공한 피드백도, 실패한 피드백도 기록
   → "이 변경이 왜 효과가 있었는가?" 추적 가능
   → 메타 학습: 피드백 자체의 효과를 학습

④ 비용 $0 유지
   분석/예측 모두 로컬 LLM (qwen2.5-7b, deepseek-r1-32b)
   외부 API 의존 없는 자립적 피드백 시스템

⑤ 일일성장 환류 사이클과 연계
   데이터축적 → 분석리포트 → 피드백대응 → 반복
   시그마팀이 이 사이클의 엔진 역할
   적용 순서: 제이→루나→스카→클로드→블로→워커→에디→연구→감정
```

## 9. Layer 3: 시그마팀 자체 피드백 (메타 학습)

시그마팀이 다른 팀을 분석하고 피드백하는 것만으로는 불완전하다.
**시그마팀 자신의 분석/피드백 품질도 평가하고 개선**해야 한다.

```
Layer 1: 팀 내부 자체 학습 — "내가 잘했나?"
Layer 2: 시그마팀 크로스팀 분석 — "전체적으로 뭐가 문제지?"
Layer 3: 시그마팀 자체 피드백 — "내 분석이 맞았나? 내 피드백이 효과 있었나?"
```

### 9-1. 시그마팀 자체 피드백 대상

```
① 분석 정확도 피드백
  "hawk가 경고한 리스크가 실제로 발생했는가?"
  "dove가 추천한 확대 전략이 실제 성과 개선으로 이어졌는가?"
  "owl이 감지한 추세가 실제로 이어졌는가?"
  → 예측/분석이 틀렸다면 → 분석 프롬프트/기준 자동 보정!

② 피드백 효과 피드백
  "hiring-contract 점수 변경 후 실제 에이전트 성과가 개선되었는가?"
  "Standing Orders 승격 후 에러가 실제로 줄었는가?"
  "파라미터 튜닝 후 실제 수익률/조회수가 올랐는가?"
  → 피드백이 효과 없었다면 → 피드백 방법 자체를 변경!

③ 편성 효과 피드백
  "오늘 편성(hawk+루나)이 적절했는가?"
  "10팀 전부 분석한 날 vs 3팀만 집중한 날, 어느 쪽이 인사이트 더 많았는가?"
  "이번 주 owl이 빠졌는데, 장기 추세를 놓쳤는가?"
  → 편성 전략 자체를 최적화!

④ 에이전트 보강 피드백
  "현재 12명으로 충분한가? 새로운 성향이 필요한가?"
  "특정 에이전트가 계속 선택 안 되면? → 역할 재정의 또는 제거"
  "새로운 분석 관점이 필요한 도메인이 발생하면? → 에이전트 추가"
  → 시그마팀 구성 자체를 진화!

⑤ 워크플로우 방법론 피드백
  "일일 사이클(06~09시)이 최적인가? 시간대 조정 필요?"
  "분석 깊이가 적절한가? 너무 얕거나 너무 깊거나?"
  "리포트 형식이 마스터에게 유용한가?"
  → 시그마팀 운영 방식 자체를 개선!

⑥ 다른 팀 구조 피드백
  "루나팀 매매 전략을 바꿔야 하는가?"
  "블로팀 작가 편성을 조정해야 하는가?"
  "다윈팀 검색 도메인을 확대/축소해야 하는가?"
  → 다른 팀의 전략/구조 자체에 대한 제안!
```

### 9-2. 자체 피드백 주기

```
일일 (자동): 분석 정확도 추적
  → 어제 분석/예측 vs 오늘 실제 결과 비교
  → 정확도 점수 자동 계산 + RAG에 축적

주간 (금요일 owl): 피드백 효과 + 편성 효과 평가
  → 이번 주 피드백이 실제 효과가 있었는가?
  → 이번 주 편성 전략이 적절했는가?
  → 다음 주 편성 전략 조정 제안

월간 (마스터 리뷰): 에이전트 보강 + 구조 피드백
  → 시그마팀 에이전트 추가/제거 제안
  → 다른 팀 구조 변경 제안
  → 워크플로우 방법론 개선 제안
  → 마스터 승인 후 적용
```

### 9-3. 자체 피드백 핵심 원칙

```
① 시그마팀도 Layer 1처럼 자체 학습!
   분석 결과 → 실제 결과 → 분석 정확도 → 다음 분석에 반영

② 피드백의 피드백 = 메타 학습!
   피드백 적용 → 효과 측정 → 피드백 방법 자체를 개선
   → "피드백하는 방법을 학습하는" 시스템!

③ 범위가 유동적!
   때로는 피드백 방법 자체에 대한 피드백
   때로는 에이전트 보강/제거 제안
   때로는 워크플로우 방법론 개선
   때로는 루나 매매 전략 변경 제안
   때로는 전체 팀 구조 재편 제안
   → 시그마팀의 피드백 범위에 제한 없음!

④ 자기 진화 = Self-Evolving!
   시그마팀이 자기 자신을 포함한 전체 시스템을 진화시키는 엔진!
```
