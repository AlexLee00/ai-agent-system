## 2026-03-18 — Dev Vlog

오늘 코덱이 가장 먼저 정리한 문제는 “기능은 계속 늘어나는데, 세션이 바뀔 때마다 다시 맥락을 찾는 비용이 크다”는 점이었다.  
실제로 구현은 많이 올라와 있었지만, 문서 이름과 역할이 어긋나 있어서 다음 세션이 어디서부터 읽어야 하는지 선명하지 않았다.

그래서 오늘은 단순 기능 개발과 문서 정리를 따로 보지 않고, 둘을 하나의 연속성 문제로 봤다.

첫 번째로는 스카 예측 엔진을 손봤다.  
지금 스카의 핵심 문제는 모델 종류가 부족하다기보다, 기존 엔진과 다른 계열 모델을 안전하게 비교할 수 없다는 점이었다.  
그래서 기존 엔진을 교체하지 않고 `knn-shadow-v1`를 `shadow`로 붙였다. 이 방식은 운영 리스크가 낮고, 실제 매출 확정 데이터가 쌓이면 `primary vs shadow`를 비교해 더 나은 방향을 판단할 수 있다.  
중요했던 건 “바로 갈아끼우지 않는다”는 원칙이었다. 내부 MVP와 운영 안정성, 그리고 나중에 SaaS로 갈 때 tenant별 실험 구조까지 생각하면 shadow 비교가 가장 맞는 단계였다.

두 번째로는 워커 문서 흐름을 한 단계 더 닫았다.  
이전까지는 문서를 업로드하고 파싱하고, 프롬프트로 재사용하는 지점까진 있었지만, 그 문서가 실제로 어떤 업무 결과를 만들었는지는 추적이 약했다.  
오늘은 문서 상세, 재사용 이벤트, 업무 생성 결과 연결, 그리고 문서별 전환율까지 보이게 했다.  
이건 단순 UX 개선이 아니라, “AI가 실제 업무에 얼마나 유효하게 쓰였는가”를 측정하는 운영 데이터 기반을 만든 것이다.

세 번째로는 일일 운영 분석 리포트를 보수적으로 다시 잡았다.  
기존엔 자동화 런타임에서 일부 health script가 실패하면 fallback 추론이 팀 상태를 과장해서 보여줄 가능성이 있었다.  
운영 리포트는 화려한 분석보다 신뢰도가 먼저라서, 오늘은 `fallback_probe_unavailable`을 장애처럼 다루지 않도록 입력 구조를 바꿨다.  
이건 AI Ops Platform에서 아주 중요한 태도다. 근거가 약한 자동 판단은 사람을 오히려 더 피곤하게 만든다.

그리고 마지막으로 문서체계를 손봤다.  
기존 `improvement-ideas.md`는 사실상 “아이디어 메모”가 아니라 구현 추적 문서였기 때문에 이름을 `PLATFORM_IMPLEMENTATION_TRACKER.md`로 바꿨다.  
여기에 더해서 `DOCUMENTATION_SYSTEM.md`, `DEV_LOG.md`, `DEV_VLOG.md`를 추가했다.  
오늘의 핵심은 문서를 더 만드는 것이 아니라, 문서가 각자 다른 역할을 하게 분리한 것이다.

정리하면 오늘의 고민은 세 가지였다.

- 스카는 어떻게 새 모델을 안전하게 비교할 것인가
- 워커 문서 흐름을 어떻게 실제 업무 결과와 연결할 것인가
- 세션이 바뀌어도 개발이 왜 끊기지 않게 만들 것인가

해결 방식도 세 가지였다.

- 스카는 `shadow`
- 워커는 `reuse trace`
- 문서는 `role separation`

다음으로 볼 것은 분명하다.

- 스카 shadow 비교 데이터가 actual과 함께 쌓이기 시작하는지
- 워커 문서 재사용 품질을 실제 수정량/확정률로 더 평가할 수 있는지
- 일일 운영 분석 리포트가 자동화 런타임에서도 health 신호를 더 안정적으로 읽을 수 있는지

오늘 작업은 기능 추가이기도 했지만, 더 본질적으로는 시스템이 스스로의 맥락을 잃지 않게 만드는 작업이었다.
