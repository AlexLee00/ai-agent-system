## 1. 팀 개요

```
현재: bots/legal (package.json만 존재, "판례봇" 설명)
변경: bots/legal → 역할 확장 (리네임 없이 역할 재정의)
     "판례봇" → "감정팀 — 법원 SW 감정 자동화 + 판례 분석"

SW 감정이란:
  법원에서 소프트웨어 관련 분쟁(저작권 침해, 하자, 계약 위반 등) 발생 시
  전문가가 소스코드를 분석하여 감정서를 작성하는 업무.
  마스터(Jay)가 법원 SW 감정인으로 활동 중.

핵심 역할:
  ① 감정 유형 분류: 사건 내용 분석 → 감정 유형 자동 분류
  ② 소스코드 분석: 코드 유사도, 구조 분석, 기능 매핑
  ③ 판례 서칭: 유사 판례 검색 → 참고 자료 정리
  ④ 감정서 초안 작성: 분석 결과 → 법원 제출용 감정서 초안
  ⑤ 품질 검증: 감정서 논리 일관성, 법률 용어 정확성 검증
  ⑥ 피드백 학습: 감정 결과 → RAG 저장 → 다음 감정에 참고
```

## 2. 팀 구성 (에이전트)

```
팀장: 저스틴 (Justin) — 감정 총괄, 사건 배정, 최종 검토
  역할: 감정 요청 수신 → 유형 분류 → 에이전트 배정 → 최종 감정서 검토
  모델: anthropic (고품질 판단, 법률 정확성 요구)

에이전트 1: 브리핑 (Briefing) — 사건분석 + 감정요청 분석 + 문서 작성 전문
  역할: 감정 촉탁서 분석, 사건/감정소요 분석, 계획서/질의서/보고서 작성
  모델: anthropic (법률 문서 분석 + 작성, 정확한 용어)
  기능:
    → 감정 촉탁서 분석: 사건번호, 당사자, 감정사항 추출
    → 감정소요 분석: 필요 인력, 기간, 비용, 기술 범위 산출
    → 감정착수계획서 작성: 감정 일정, 방법론, 인력 배치
    → 질의서 작성: 1차/2차 질의서 항목 도출
    → 현장실사계획서 작성: 실사 일정, 확인 항목, SW 기능 분류 기준
    → 감정보고서 작성: 최종 감정 결과 (법원 양식)
  감정서 구조:
    1) 감정 개요 (사건번호, 당사자, 감정사항)
    2) 분석 방법론 (사용 도구, 분석 기준)
    3) 사건 및 감정소요 분석 결과
    4) 질의/현장확인 결과 정리
    5) 현장실사 결과 (기능별 가동/부분가동/불가동)
    6) 판례 참조 (유사 사건 선례)
    7) 감정 의견 (결론 + 근거)
    8) 첨부 자료 (코드 비교표, 스크린샷, 기능 분류표)

에이전트 2: 렌즈 (Lens) — 소스코드 분석 전문
  역할: 소스코드 유사도 분석, 구조 분석, 기능 매핑, 복사 여부 판단
  모델: claude-code/sonnet (코드 분석에 최적)
  분석 유형:
    → 코드 유사도: 문자열/토큰/AST 기반 유사도 측정
    → 구조 분석: 아키텍처 패턴, 모듈 의존성, 클래스 계층
    → 기능 매핑: 원고 코드 기능 ↔ 피고 코드 기능 1:1 대응
    → 복사 탐지: 변수명 변경, 주석 제거, 순서 변경 등 난독화 감지
  감정 유형별 특화:
    웹: HTML/CSS/JS 프론트엔드 + 백엔드 API + DB 스키마
    앱: iOS/Android 네이티브 + 크로스플랫폼 (Flutter/RN)
    DB: 스키마 설계, 쿼리 패턴, 저장 프로시저
    임베디드: 펌웨어, 디바이스 드라이버, RTOS
  출력: 코드 분석 리포트 (유사도 %, 매핑 테이블, 증거 스크린샷)

에이전트 3: 가람 (Garam) — 국내 판례 서칭 전문
  역할: 국내 유사 판례 검색 + 법률 근거 정리 + 선례 분석
  이름 유래: 가람(강) — 한국 법률의 흐름을 읽는다
  모델: anthropic (법률 문서 분석)
  소스:
    → 대법원 종합법률정보 (판례 검색)
    → 하급심 판례 DB
    → 한국소프트웨어감정평가학회 자료
    → 한국저작권위원회 DB
  출력: 국내 판례 리포트 (유사 사건, 판결 요지, 적용 법률)

에이전트 4: 아틀라스 (Atlas) — 해외 판례 서칭 전문
  역할: 해외 유사 판례 검색 + 국제 법률 동향 + 비교법적 분석
  이름 유래: 아틀라스 — 세계를 아우르는 법률 지식
  모델: anthropic (영문 법률 문서 분석)
  소스:
    → US Copyright Office / USPTO
    → EU Court of Justice (CURIA)
    → WIPO (세계지적재산권기구)
    → IEEE/ACM 학술 논문 (SW 감정 방법론)
    → GitHub DMCA 사례
  출력: 해외 판례 리포트 (국제 동향, 비교법적 시사점)

에이전트 5: 클레임 (Claim) — 원고 자료 분석 전문
  역할: 원고 측 제출 자료(소스코드, 주장서면, 증거) 심층 분석
  모델: claude-code/sonnet (코드 분석) + anthropic (서면 분석)
  분석 항목:
    → 원고 소스코드 구조/기능/특징 분석
    → 원고 주장의 기술적 타당성 검증
    → 원고 제출 증거 자료 정리 + 증거력 평가
    → 원고 측 핵심 쟁점 도출
  출력: 원고 자료 분석 리포트 (구조도, 기능 목록, 쟁점 정리)
  원칙: 객관적 분석 (원고 편향 금지)

에이전트 6: 디펜스 (Defense) — 피고 자료 분석 전문
  역할: 피고 측 제출 자료(소스코드, 답변서, 증거) 심층 분석
  모델: claude-code/sonnet (코드 분석) + anthropic (서면 분석)
  분석 항목:
    → 피고 소스코드 구조/기능/특징 분석
    → 피고 주장의 기술적 타당성 검증
    → 피고 제출 증거 자료 정리 + 증거력 평가
    → 피고 측 핵심 쟁점 도출
  출력: 피고 자료 분석 리포트 (구조도, 기능 목록, 쟁점 정리)
  원칙: 객관적 분석 (피고 편향 금지)

  클레임 + 디펜스 연동:
    → 양측 분석 완료 후 렌즈(코드 분석)에게 비교 분석 요청
    → 양측 주장의 기술적 교차 검증
    → 쟁점별 양측 주장 대비표 작성

에이전트 7: 퀼 (Quill) — 감정서 초안 작성 전문
  역할: 분석 결과를 법원 제출용 감정서 형식으로 작성
  이름 유래: 퀼(깃펜) — 법률 문서의 전통적 상징
  모델: anthropic (법률 문서 작성, 정확한 용어)
  입력: 렌즈 분석 + 클레임/디펜스 양측 분석 + 가람/아틀라스 판례
  출력: 감정서 초안 (법원 양식 준수)

에이전트 8: 밸런스 (Balance) — 감정서 품질 검증
  역할: 감정서 초안의 논리 일관성, 법률 용어 정확성, 증거 충분성 검증
  이름 유래: 밸런스(저울) — 양측 균형 잡힌 공정한 검증

에이전트 9: 컨트랙트 (Contract) — 계약서 검토 전문
  역할: SW 개발 계약서, 유지보수 계약서, SLA 등 계약 문서 분석
  모델: anthropic (법률 문서 분석, 계약 조항 해석)
  기능:
    → 계약서 조항 분석: 개발 범위, 납품 기준, 하자 담보, 지체상금 등
    → 계약 위반 여부 판단: 원고/피고 주장과 계약 조항 대비
    → SLA/KPI 분석: 가용성, 응답시간, 장애복구 기준 충족 여부
    → 계약 해석 쟁점 도출: 모호한 조항, 누락된 조항, 해석 차이
    → 손해배상 산정 근거: 계약상 손해배상 조항 + 실제 손해 대비
  입력: 계약서 원문 + 사건 쟁점 + 브리핑의 사건분석 결과
  출력: 계약서 분석 리포트 (조항별 분석, 위반 여부, 쟁점, 손해 산정 근거)
  감정 유형별 활용:
    하자 분쟁: 하자 담보 기간/범위 + 납품 검수 기준 분석
    계약 위반: 개발 범위 vs 실제 납품물 대비
    대금 분쟁: 기성금/잔금 지급 조건 + 지체상금 계산
  모델: anthropic (비판적 분석)
  검증 항목:
    → 논리 일관성: 분석 결과와 결론이 일치하는가
    → 법률 정확성: 법률 용어, 조항 인용이 정확한가
    → 증거 충분성: 결론을 뒷받침하는 증거가 충분한가
    → 중립성: 한쪽에 치우치지 않는 객관적 서술인가
    → 형식 준수: 법원 감정서 양식에 맞는가
  출력: 리뷰 리포트 (수정 필요 항목, 보완 제안)
  원칙: 퀼과 독립적 판단 (작성자 ≠ 검증자)
```

### 에이전트별 분석 방법론 상세

```
렌즈 (소스코드 분석):
  유사도 분석 3중 방법:
    ① 문자열 유사도: diff 기반, 라인 단위 일치율
    ② 토큰 유사도: AST 파싱 → 토큰 시퀀스 비교 (변수명 무시)
    ③ 구조 유사도: 클래스/함수/모듈 계층 비교
  복사 탐지 기법:
    → 변수명 변경 탐지 (AST 구조 동일 + 이름만 다름)
    → 주석 제거/추가 탐지
    → 코드 순서 변경 탐지 (함수 재배치)
    → 난독화 역공학 (minify/obfuscate 역추적)
  기능 매핑:
    → 원고 기능 목록 ↔ 피고 기능 목록 1:1 대응표 생성
    → 동일 기능의 구현 방식 비교 (알고리즘, 라이브러리, 설계 패턴)
  출력: 유사도 %, 매핑 테이블, 증거 스크린샷, 비교 코드 블록

클레임 (원고 자료 분석):
  소스코드 분석:
    → 프로젝트 구조 파악 (디렉토리, 모듈, 의존성)
    → 핵심 기능 목록 추출 (API, UI, 비즈니스 로직)
    → 코드 품질 메트릭 (복잡도, 중복률, 테스트 커버리지)
    → 개발 이력 분석 (git log, 커밋 패턴, 기여자)
  서면 분석:
    → 원고 주장의 기술적 타당성 검증
    → 주장 근거와 실제 코드의 일치 여부
    → 과장/누락된 주장 식별
  증거 평가:
    → 제출 증거의 증거력 등급 (직접/간접/추정)
    → 증거 간 모순 여부 체크

디펜스 (피고 자료 분석):
  (클레임과 동일 분석 프레임워크, 피고 측에 적용)
  추가 분석:
    → 독자적 개발 증거 확인 (개발 환경, 커밋 이력, 설계 문서)
    → 오픈소스/공통 라이브러리 사용 여부 (유사도 원인 분석)
    → 피고 측 반론의 기술적 타당성 검증

가람 (국내 판례 서칭):
  검색 전략:
    → 키워드: 사건 유형 + 기술 분야 + 법률 조항
    → 대법원 → 고등법원 → 지방법원 순 우선순위
    → 최신 판례 우선 (5년 이내)
  분석 방법:
    → 판결 요지에서 감정 관련 핵심 판단 추출
    → 법원이 인정한 감정 방법론 파악
    → 현재 사건에 적용 가능한 법리 정리
  출력: 판례 5건 이내, 각 판례별 (사건번호, 요지, 적용 법률, 시사점)

아틀라스 (해외 판례 서칭):
  검색 전략:
    → US: Copyright Office + USPTO + federal court records
    → EU: CURIA + EPO + national courts
    → 국제: WIPO 중재/조정 사례 + ICC
  분석 방법:
    → 비교법적 시사점 (한국법 대비 차이점)
    → 국제 동향이 국내 판결에 미칠 영향
    → 글로벌 기술 분쟁 트렌드
  출력: 해외 판례 3건 이내 + 비교법적 분석 요약
```

## 4. DB 스키마 (PostgreSQL, legal 스키마)

```sql
-- 감정 사건
CREATE TABLE legal.cases (
  id SERIAL PRIMARY KEY,
  case_number TEXT NOT NULL,          -- 법원 사건번호
  court TEXT,                         -- 관할 법원
  case_type TEXT,                     -- copyright/defect/contract/trade_secret
  plaintiff TEXT,                     -- 원고
  defendant TEXT,                     -- 피고
  appraisal_items JSONB,             -- 감정사항 목록
  status TEXT DEFAULT 'received',     -- received/analyzing/drafting/reviewing/completed
  assigned_agents JSONB,              -- 배정된 에이전트 목록
  deadline DATE,                      -- 감정서 제출 기한
  created_at TIMESTAMPTZ DEFAULT NOW(),
  completed_at TIMESTAMPTZ
);

-- 코드 분석 결과
CREATE TABLE legal.code_analyses (
  id SERIAL PRIMARY KEY,
  case_id INTEGER REFERENCES legal.cases(id),
  analysis_type TEXT,                 -- similarity/structure/function_mapping/copy_detection
  source_type TEXT,                   -- web/app/db/embedded
  similarity_score NUMERIC(5,2),     -- 유사도 % (0~100)
  mapping_data JSONB,                -- 기능 매핑 테이블
  evidence JSONB,                    -- 증거 (파일명, 라인번호, 스크린샷 경로)
  conclusion TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 판례 검색 결과
CREATE TABLE legal.case_references (
  id SERIAL PRIMARY KEY,
  case_id INTEGER REFERENCES legal.cases(id),
  ref_case_number TEXT,              -- 참조 판례 사건번호
  court TEXT,                        -- 판결 법원
  decision_date DATE,                -- 판결일
  summary TEXT,                      -- 판결 요지
  applicable_law TEXT,               -- 적용 법률/조항
  relevance_score NUMERIC(3,1),      -- 관련성 점수
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 감정서 버전 관리
CREATE TABLE legal.reports (
  id SERIAL PRIMARY KEY,
  case_id INTEGER REFERENCES legal.cases(id),
  version INTEGER DEFAULT 1,
  content_path TEXT,                  -- 감정서 파일 경로
  review_status TEXT DEFAULT 'draft', -- draft/reviewed/approved/submitted
  review_notes TEXT,                  -- 밸런스 피드백
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

## 5. 핵심 워크플로우

### 5-1. 감정 프로세스 — 실제 SW 감정 워크플로우

```
[1단계: 감정 촉탁]
  법원에서 감정 촉탁서 수신 (마스터)
  → 저스틴(팀장)에게 전달
  → legal.cases INSERT (사건번호, 당사자, 감정사항)
  ↓
[2단계: 사건 및 감정소요 분석] 브리핑
  → 촉탁서 분석: 사건 쟁점, 감정 범위, 기술 분야 파악
  → 감정소요 산출: 필요 인력, 예상 기간, 비용, 기술적 난이도
  → 감정 유형 분류 (저작권/하자/계약/영업비밀)
  ↓
[2.5단계: 양측 자료 분석] 클레임 + 디펜스 (병렬)
  → 클레임: 원고 소스코드/주장서면/증거 심층 분석
  → 디펜스: 피고 소스코드/답변서/증거 심층 분석
  → 양측 분석 완료 → 렌즈에게 비교 분석 요청
  → 쟁점별 양측 주장 대비표 작성
  ↓
[3단계: 국내/해외 판례 분석] 가람 + 아틀라스 (병렬)
  → 국내: 대법원/하급심 유사 판례 + 한국저작권위원회 자료
  → 해외: US/EU/WIPO 유사 판례 + 국제 동향
  → 비교법적 시사점 정리
  ↓
[4단계: 감정계획 작성] 브리핑
  → 감정 방법론, 일정, 분석 기준, 에이전트 배정 결정
  → 저스틴 승인
  ↓
[5단계: 감정착수계획서 발송] 브리핑 + 저스틴
  → 감정착수계획서 작성 (법원 양식)
  → 마스터 최종 검토 → 법원에 발송
  ↓
[6단계: 1차 질의서 발송] 브리핑
  → 사건 분석 기반 질의서 항목 도출
  → 원고/피고에게 확인이 필요한 사항 정리
  → 마스터 검토 → 법원 경유 당사자에게 발송
  ↓
[7단계: 1차 현장확인(인터뷰)]
  → 마스터가 현장 방문 / 온라인 인터뷰 진행
  → 인터뷰 기록 → 시스템에 입력
  → 브리핑이 인터뷰 결과 분석 + 정리
  ↓
[8단계: 2차 질의서 발송] 브리핑
  → 1차 현장확인에서 추가 확인 필요 사항 도출
  → 마스터 검토 → 법원 경유 발송
  ↓
[9단계: 2차 현장확인(인터뷰)]
  → 마스터가 추가 현장 방문 / 인터뷰
  → 인터뷰 기록 입력 + 분석
  ↓
[10단계: 현장실사계획서 발송] 브리핑
  → 실사 일정, 확인 항목, SW 기능 분류 기준 작성
  → 마스터 검토 → 법원 경유 발송
  ↓
[11단계: 현장실사] 렌즈 + 브리핑
  → 마스터가 현장에서 소프트웨어 기능 확인
  → SW 기능 3단계 분류:
    1스텝: 대분류 (모듈/시스템 단위)
    2스텝: 중분류 (기능 단위)
    3스텝: 소분류 (세부 기능/화면 단위)
  → 각 기능별 판정:
    ✅ 가동: 정상 작동
    ⚠️ 부분가동: 일부 기능만 작동
    ❌ 불가동: 작동하지 않음
  → 렌즈: 소스코드 유사도/구조 분석 (필요 시)
  → 기능 분류표 + 가동 판정표 생성
  ↓
[12단계: 감정보고서 작성] 퀼 + 밸런스 + 저스틴
  → 브리핑: 전체 프로세스 결과 종합 → 감정보고서 초안
  → 밸런스: 논리/법률/증거/중립성 검증
  → 저스틴: 최종 검토
  → 마스터: 최종 수정 + 서명 → 법원 제출
  ↓
[13단계: 피드백] 저스틴
  → 감정 결과 → RAG 대도서관 저장 (rag_legal 컬렉션)
  → 법원 판결 수신 시 → 감정 정확도 기록 → 다음 감정에 반영
```

## 6. 다른 팀과의 연동

```
감정팀 → 연구팀:
  Scout-Legal이 SW감정 관련 최신 기법/판례 서칭 → 감정팀에 제공
  리서처가 새 감정 방법론 심층 연구 → 렌즈에 반영

감정팀 → 데이터 사이언스 팀:
  감정 사건 데이터 축적 → 유형별 패턴 분석
  유사도 측정 정확도 개선 데이터 제공

감정팀 → 대도서관:
  rag_legal 컬렉션: 감정 사례 + 판례 + 법원 판결 결과
  다른 팀 에이전트가 법률 관련 질문 시 참조 가능
```

## 8. 안전 원칙

```
① 감정 데이터 보안
  → 소스코드는 cases/ 디렉토리에만 저장
  → GitHub에 push 금지 (.gitignore에 cases/ 추가)
  → 사건 완료 후 소스코드 삭제 (마스터 수동)

② 감정서는 반드시 마스터 최종 검토
  → 에이전트가 작성한 초안은 "초안"일 뿐
  → 마스터가 최종 수정 + 서명 후 법원 제출
  → 에이전트가 직접 법원에 제출하는 일 없음

③ 중립성 원칙
  → 프롬프트에 "객관적 분석" 강조
  → 원고/피고 어느 쪽에도 치우치지 않는 서술
  → 밸런스가 중립성 검증 항목 포함

④ 법률 용어 정확성
  → context/LEGAL_TERMS.md 법률 용어 사전 참조
  → 잘못된 법률 용어 사용 시 밸런스가 차단
```
